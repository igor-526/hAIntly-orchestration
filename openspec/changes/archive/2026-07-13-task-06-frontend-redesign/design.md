## Context

`main-fe` — Next.js App Router приложение, в котором auth-форма частично оформлена utility-классами, а защищённая рабочая область использует большой набор самописных глобальных CSS-компонентов. Управление HH-аккаунтами сейчас открывается как dropdown внутри шапки и разрешает удаление только активного аккаунта; профильное меню содержит нерабочие пункты. При этом `main-be` уже предоставляет все необходимые cookie-auth и HH account proxy операции, включая `POST /api/auth/logout`, поэтому изменение является frontend-only.

Владельцем auth-сессии остаётся `main-be`, владельцем HH-связей — `profile-service`; браузер инициирует HTTP-запросы только к `main-be` с cookie credentials. OAuth popup/callback продолжает использовать существующий HTTP-поток через `main-be`. API, SSE, NATS JetStream, БД, фоновые процессы и персональные данные этим change не изменяются.

## Goals / Non-Goals

**Goals:**

- создать единый MUI visual foundation для всех маршрутов и служебных состояний `main-fe`;
- автоматически следовать системной light/dark теме без хранения пользовательского выбора;
- сохранить текущую feature-архитектуру, сетевые контракты и обязательные UI-состояния;
- улучшить `/login` через доступный эксклюзивный `ToggleButtonGroup`;
- вынести полный цикл управления HH-аккаунтами в верхний Drawer, включая удаление неактивного аккаунта;
- реализовать корректный logout и сохранить «Настройки» видимыми, но недоступными.

**Non-Goals:**

- изменение endpoint, DTO, cookie-политики или сервисных границ;
- прямые вызовы `profile-service`, HH либо других профильных сервисов из браузера;
- ручной переключатель темы или сохранение темы в local/session storage;
- реализация настроек, вакансий, резюме и AI-действий;
- изменение OAuth popup/callback orchestration.

## Decisions

### 1. Один корневой MUI provider для App Router

В корневом layout будет подключён клиентский UI provider, который объединяет официальный MUI cache provider для совместимой версии Next.js App Router, `ThemeProvider`, `CssBaseline` и существующий `AuthProvider`. Зависимости `@mui/material`, Emotion и официальный пакет интеграции Next.js фиксируются в `package.json` и lock-файле. Страницы, dialogs, menus, loading/error/empty states и workspace shell переводятся на MUI-компоненты; глобальный CSS остаётся только для действительно глобальных правил, которые не выражаются темой или `sx`.

Альтернатива — сохранить самописный CSS и точечно добавить Drawer/ToggleButton — отклонена: она оставляет два визуальных набора компонентов и не выполняет требование переписать страницы на библиотеку.

### 2. Тема выводится только из `prefers-color-scheme`

UI provider использует системное media query и строит светлую либо тёмную MUI palette. Изменение настройки ОС во время открытой страницы обновляет тему без reload. Выбор не записывается в cookies, local storage или session storage. Начальный рендер и hydration должны использовать поддерживаемый MUI механизм color-scheme/media query так, чтобы не показывать устойчиво неверную палитру и не создавать hydration errors; в тестах `matchMedia` контролируется детерминированно.

Альтернатива — CSS-only `prefers-color-scheme` — отклонена, поскольку MUI-компоненты и их состояния должны получать согласованные palette tokens. Ручной toggle исключён исходным scope.

### 3. `/login` использует эксклюзивный ToggleButtonGroup

Существующий `useLogin` сохраняет orchestration формы, валидацию и сетевой слой. UI заменяется на MUI `Paper`, `Stack`, `TextField`, `Alert`, `Button` и полноширинный `ToggleButtonGroup` с двумя значениями. Обработчик игнорирует `null`, поэтому повторное нажатие не снимает выбранный режим. Переключение продолжает вызывать существующий `switchMode`, который очищает пароль и ошибки.

Альтернатива — Tabs — отклонена, потому что ТЗ явно выбирает Toggle Button, а режимы являются взаимоисключающими действиями формы.

### 4. Верхний Drawer владеет представлением управления HH-аккаунтами

Кнопка в шапке показывает активный HH-аккаунт и открывает MUI `Drawer` с `anchor="top"`. Drawer отображает все связи, явно отмечает активную, позволяет выбрать любую другую, запустить существующий OAuth connect и запросить удаление у каждой отдельной строки. Цель удаления хранится отдельно от active id, а MUI `Dialog` показывает имя именно выбранной связи. Удаление выполняется только после подтверждения; pending блокирует повтор и опасные конкурирующие действия. После успеха используется существующая повторная загрузка списка/active id; при ошибке dialog остаётся доступным для повтора или отмены. Фокус, Escape, backdrop и keyboard navigation предоставляются MUI primitives и проверяются component tests.

Альтернатива — dropdown в шапке — отклонена из-за недостаточного пространства для независимых действий над несколькими аккаунтами. Постоянный sidebar также отклонён: он уменьшает рабочую область и не соответствует требованию верхней панели.

### 5. Logout расширяет существующую auth orchestration

`authService` добавляет вызов существующего `POST /api/auth/logout`; `AuthProvider` предоставляет единое действие logout и состояния pending/error. Успешный ответ очищает клиентского пользователя, устанавливает `anonymous` и приводит защищённый route gate к замене маршрута на `/login`. Пока запрос выполняется, повторный logout заблокирован. При сетевой/серверной ошибке текущая сессия не объявляется завершённой, пользователь получает безопасную ошибку и может повторить действие. Cookie и токены не читаются, не сохраняются и не логируются frontend.

Альтернатива — только локально очистить React state — отклонена: HttpOnly cookie останется действующей и следующая verify восстановит сессию.

### 6. Тестирование и runtime-проверка остаются автономными на unit-уровне

Component tests проверяют обе media-query палитры и изменение системной темы, эксклюзивное переключение login modes, открытие/закрытие Drawer, выбор аккаунта, удаление активного и неактивного аккаунта, ошибки/pending, disabled settings и logout. Service/provider tests проверяют сериализацию logout и переходы auth state без live backend. После тестов и lint frontend-agent обязан выполнить `make fe-build`, `make fe`, проверить frontend-порт и логи контейнера; Quality Gate повторяет применимые проверки отдельно.

## Risks / Trade-offs

- [Несовпадение server/client palette при hydration] → использовать официальный App Router integration и поддерживаемый MUI механизм системной темы; добавить regression test без hydration warning.
- [Рост client bundle из-за MUI и Emotion] → импортировать компоненты модульно, не добавлять второй icon/component framework и проверить production build.
- [Регрессия доступности при переносе самописных controls] → опираться на MUI semantics, сохранять доступные labels и тестировать keyboard/focus flows.
- [Удаление неверного HH-аккаунта] → хранить явный target id из строки Drawer и показывать имя цели в отдельном confirmation dialog.
- [UI считает logout успешным при ошибке backend] → переводить auth state в `anonymous` только после успешного `204`, иначе сохранять сессию и показывать retryable error.
- [Системная тема недоступна в тестовом DOM] → определить общий управляемый `matchMedia` mock в test setup.

## Migration Plan

1. Добавить MUI/Emotion/App Router зависимости и корневой provider, сохранив работающий auth provider.
2. Перевести общие route states и `/login` на MUI, затем перенести workspace/header/Drawer/Dialog.
3. Добавить logout в существующие service/provider boundaries и подключить его к профильному меню.
4. Обновить автономные тесты и удалить ставшие неиспользуемыми компонентные CSS-правила только после миграции всех страниц.
5. Пересобрать и поднять `main-fe`, проверить light/dark, responsive UI, logout и HH account flows.

Rollback выполняется возвратом frontend commit и предыдущего lock-файла/образа; миграций данных и координированного backend rollback нет.
