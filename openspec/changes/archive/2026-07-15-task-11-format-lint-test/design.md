## Context

В репозитории уже есть `Makefile` у `main-be`, `profile-service`, `vacancy-service`, `ai-service`, `main-fe` и `fastapi_template`. Backend-цели используют Ruff для изменения форматирования, mypy и Flake8 для lint и pytest либо сервисный test script; frontend предоставляет test и lint, но цель format пуста. Профили агентов перечисляют отдельные низкоуровневые команды, поэтому единый Makefile-интерфейс не является обязательной границей передачи и независимо не воспроизводится Quality Gate.

Изменение пересекает несколько сервисов и workflow-документы, но остаётся локальным контуром разработки. Оно не создаёт runtime-потоков: нет владельца продуктовых данных, инициатора или получателя по HTTP, SSE либо NATS JetStream. API, события, БД, фоновые процессы, auth, `X-User-Id`, service credentials, HH/AI-интеграции, безопасность runtime и обработка персональных данных не меняются.

## Goals / Non-Goals

**Goals:**

- Сделать `make format`, `make lint` и `make test` единым интерфейсом качества во всех существующих основных сервисах и `fastapi_template`.
- Реализовать воспроизводимое frontend-форматирование через formatter, объявленный в development-зависимостях проекта.
- Исключить `.github` и `.helm` из области любого изменяющего форматирования независимо от того, существуют каталоги сейчас или появятся позже.
- Исправить реальные нарушения кода, конфигурации и тестов так, чтобы три команды успешно проходили в каждом затронутом сервисе.
- Закрепить одинаковые обязательства профильных исполнителей и независимого Quality Gate.

**Non-Goals:**

- Изменение пользовательского поведения, runtime API, HTTP/SSE/NATS-контрактов, схем БД или сервисных границ.
- Унификация конкретных внутренних инструментов backend и frontend, если Makefile-контракт и требования к исключениям соблюдены.
- Создание `notification-service`, изменение общей инфраструктуры или форматирование `.github`/`.helm`.
- Массовое переписывание кода вне исправлений, необходимых для успешных format/lint/test.

## Decisions

### 1. Makefile является стабильной точкой входа

В каждом затронутом сервисе и `fastapi_template` цели `format`, `lint`, `test` объявляются phony и выполняют полный применимый pipeline данного проекта. Агенты вызывают цели из каталога сервиса, поэтому конкретные npm/uv-команды остаются инкапсулированы рядом с manifest.

Альтернатива — закрепить в профилях прямые команды каждого инструмента. Она отклонена: такие списки расходятся с Makefile и требуют обновлять workflow при каждой замене инструмента.

### 2. Format является изменяющей и идемпотентной операцией

`make format` применяет автоматические безопасные исправления и форматирование, после чего повторный запуск не создаёт diff. Lint и test не подменяют format и завершаются ненулевым кодом при нарушении. Для frontend formatter фиксируется в `package.json`/lock-файле и вызывается через package manager, чтобы версия была воспроизводимой.

Альтернатива — сделать format только проверкой. Она отклонена, поскольку исходное назначение команды — привести файлы к согласованному виду, а проверяемость обеспечивается последующими lint/test и отсутствием diff после повторного запуска.

### 3. Исключения задаются в конфигурации инструмента и проверяются регрессией

`.github` и `.helm` исключаются в конфигурации или ignore-файле каждого formatter, а не только shell-glob в текущем Makefile. Исключение должно работать для вложенных каталогов с этими именами и после их будущего создания. Проверка создаёт или использует безопасные fixtures/временные файлы, запускает format и подтверждает, что содержимое не изменилось.

Альтернатива — полагаться на текущий набор расширений или отсутствие каталогов. Она отклонена: область formatter может расшириться, а каталоги предусмотрены как будущие.

### 4. Исправления ограничиваются качеством кода

Красные команды исправляются в исходниках, тестах и tooling-конфигурации без ослабления правил, массовых исключений исходного кода или пропуска тестов. Если исправление требует изменения продуктового поведения либо контракта, реализация останавливается для обновления OpenSpec.

### 5. Передача и review используют одинаковую матрицу

Backend Agent и Frontend Agent запускают все три Makefile-цели в каждом изменённом ими сервисе непосредственно перед передачей. Quality Gate независимо повторяет матрицу для каждого сервиса, затронутого change, и дополнительно сохраняет действующие build/runtime-проверки. Ошибка, отсутствующая цель или пропуск команды блокируют передачу либо дают `НА ДОРАБОТКУ`.

## Risks / Trade-offs

- [Автоформатирование создаст большой diff] → форматировать только область, принадлежащую сервису, отделять механические изменения от функциональных и проверять итог lint/test.
- [Разные инструменты будут по-разному трактовать ignore] → настроить исключения нативно для каждого formatter и проверить неизменность fixtures после фактического `make format`.
- [Исправление lint случайно изменит поведение] → ограничить scope, потребовать успешные тесты и независимый просмотр diff в Quality Gate.
- [Полный pipeline займёт больше времени] → сохранить единый обязательный интерфейс; оптимизация времени не должна пропускать проверки.
- [Шаблон и созданные сервисы снова разойдутся] → обновить `fastapi_template` одновременно и закрепить контракт нормативной спецификацией.

## Migration Plan

1. Обновить Makefile и formatter-конфигурации backend-сервисов и `fastapi_template`, добавить проверки исключений.
2. Добавить frontend formatter, ignore-конфигурацию, цель `make format` и lock-файл.
3. Последовательно запустить format, исправить lint/test во всех затронутых проектах и подтвердить идемпотентность format.
4. Обновить три агентских профиля на Makefile-матрицу.
5. Пересобрать и поднять изменённые контейнеры, выполнить действующие health/runtime/log проверки, затем передать отдельному Quality Gate.

Откат выполняется возвратом tooling-конфигураций, Makefile, lock-файлов, профильных документов и механических исправлений; миграции данных и координация runtime-развёртывания не нужны.

## Open Questions

Нет. Конкретный frontend formatter и минимальные совместимые обновления development-зависимостей выбираются исполнителем по текущему package ecosystem при сохранении требований спецификации.
