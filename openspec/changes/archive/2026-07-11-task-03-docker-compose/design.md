## Context

Локальная оркестрация разделена на Compose-файлы в `.docker-compose` и запускается целями корневого `Makefile`. Сейчас PostgreSQL проверяет переменную с опечаткой (`OSTGRES_DB`), а frontend получает порт через несколько несогласованных параметров: build argument `PORT`, runtime `ENV PORT`, объявленный Dockerfile порт и публикацию `${OUR_PORT}:80`. Одновременно Dockerfile использует `npm install`, хотя в репозитории есть lock-файл, а дерево frontend-зависимостей приводит к предупреждению о deprecated `whatwg-encoding`.

Изменение затрагивает локальный deployment boundary и сборку `main-fe`, но не меняет архитектурные границы HAIntly. Единственный прикладной поток frontend остаётся прежним: владелец пользовательских данных — профильный backend-сервис, инициатор HTTP/SSE-взаимодействия — `main-fe`, получатель/источник событий — `main-be`, протоколы — HTTP и SSE. Этот change не меняет адреса, payload или владение данными данного потока. NATS JetStream, HH и AI/LLM в изменяемом локальном потоке не участвуют.

## Goals / Non-Goals

**Goals:**

- Обеспечить корректный healthcheck локального PostgreSQL на основе фактических `POSTGRES_USER` и `POSTGRES_DB`.
- Сделать одну переменную `PORT` из frontend `.env` источником runtime-порта Next.js и host/container port mapping.
- Получать воспроизводимую frontend-сборку по lock-файлу без deprecated `whatwg-encoding` и без npm update notice.
- Автоматически проверять разрешённую Compose-конфигурацию и сборку.

**Non-Goals:**

- Production deployment, контейнеризация отсутствующих сервисов и изменение внешних портов backend-инфраструктуры.
- Изменение HTTP/SSE API, NATS JetStream subjects/events, схем или содержимого БД.
- Изменение фоновой обработки, HH/AI-адаптеров, безопасности, аутентификации или персональных данных.
- Безусловное обновление всех npm-пакетов до последних major-версий без проверки совместимости.

## Decisions

### 1. Healthcheck использует переменные, переданные самому PostgreSQL

Команда `pg_isready` будет обращаться к `$POSTGRES_USER` и `$POSTGRES_DB` внутри контейнера с корректным экранированием Compose. Так проверяется именно настроенная база, а readiness не зависит от дублирующих значений в YAML.

Альтернатива — записать имя пользователя и БД строковыми литералами. Она отклонена, поскольку расходится с `.env` при переименовании и создаёт второй источник конфигурации.

### 2. `PORT` является единым контрактом frontend-порта

`Makefile` продолжит передавать `services/main-fe/.env` через `--env-file`. Compose передаст `PORT` как build/runtime configuration и опубликует `${PORT}:${PORT}` с документированным значением по умолчанию. Dockerfile сохранит `ENV PORT` и `EXPOSE` на том же значении, поэтому Next.js слушает тот порт, который публикует Compose. `NEXT_PUBLIC_API_BASE_URL` остаётся отдельным build-time параметром.

Альтернатива — всегда слушать фиксированный порт внутри контейнера и менять только host port. Она технически допустима, но не выполняет явное требование о том, что образ должен открывать порт из `.env`, и сохраняет разные значения в конфигурации.

### 3. Версии зависимостей обновляются через lock-файл и проверяются фактическим деревом

Прямые frontend-зависимости обновляются только до совместимого набора; lock-файл пересоздаётся штатным npm resolver. Критерий результата — `npm ls whatwg-encoding` не показывает установленный deprecated-пакет, а тесты и production build проходят. Docker-сборка использует `npm ci`, чтобы точно воспроизводить lock-файл. Уведомление npm отключается конфигурацией npm в образе, а не глобальной установкой новой major-версии CLI.

Альтернатива — установить npm 12 глобально. Она отклонена: это увеличивает поверхность и время сборки, может быть несовместимо с выбранным Node image и не устраняет deprecated транзитивную зависимость.

### 4. Проверки выполняются на двух уровнях

Статическая проверка вызывает `docker compose config` с `PORT=3101` и подтверждает итоговое сопоставление `3101:3101` и корректную команду healthcheck. Интеграционная проверка собирает frontend-образ и поднимает инфраструктуру, после чего проверяет состояние PostgreSQL и доступность frontend на заданном host port. Если среда Quality Gate не предоставляет Docker daemon, интеграционная часть отмечается как недоступная с сохранением статических проверок.

## Risks / Trade-offs

- [Обновление транзитивных зависимостей меняет поведение тестового DOM или lint-инструментов] → Обновлять совместимый набор, проверить `npm test`, lint и production build, зафиксировать lock-файл.
- [Одинаковый host/container port может конфликтовать с локально занятым портом] → Порт остаётся настраиваемым через `.env`; конфликт должен завершать запуск явной ошибкой Docker.
- [Существующий PostgreSQL volume содержит параметры от старой конфигурации] → Healthcheck проверяет текущие настройки контейнера; при несовместимой смене credentials пользователь отдельно пересоздаёт локальный volume, автоматическое удаление данных запрещено.
- [Отключение npm notice может скрыть информацию об обновлении CLI] → Версия npm всё равно видна через `npm --version`; обновления выполняются осознанно вместе с Node base image.

## Migration Plan

1. Обновить frontend-зависимости и lock-файл, затем выполнить тесты и production build.
2. Согласовать Dockerfile, Compose и `Makefile` вокруг `PORT` и исправить healthcheck PostgreSQL.
3. Проверить разрешённую Compose-конфигурацию с `PORT=3101`, затем по возможности собрать и поднять контейнеры.
4. Откат выполняется возвратом файлов конфигурации, Dockerfile и manifest/lock-файла; volumes не удаляются и миграция данных не требуется.

## Open Questions

Нет открытых решений, блокирующих реализацию. Конкретные версии npm-зависимостей определяются во время apply по совместимости и результатам проверок, а не фиксируются заранее в design.
