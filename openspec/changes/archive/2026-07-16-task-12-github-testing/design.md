## Context

Сервисы HAIntly хранят собственные CI workflow внутри каталогов сервисов и запускают `make test` после установки зависимостей из отдельного checkout. Сейчас этот интерфейс неоднороден:

- `main-be` запускает весь pytest-набор, включая smoke, который вычисляет путь к соседнему `profile-service` и запускает его `.venv/bin/python`; такой каталог отсутствует в checkout репозитория `main-be`;
- `vacancy-service` через `scripts/test.sh` безусловно требует локальные Docker-контейнеры PostgreSQL и Redis до запуска любого теста;
- `profile-service` и `ai-service` запускают автономный pytest-набор, `main-fe` — Vitest;
- `notification-service` ещё не реализован и не имеет Makefile или тестов.

Изменение касается только маршрутизации тестов. Владельцы данных, HTTP/SSE/NATS-потоки и runtime-взаимодействия сервисов не меняются.

## Goals / Non-Goals

**Goals:**

- сделать `make test` воспроизводимым в отдельном checkout каждого реализованного проекта без Docker, доступной БД/Redis, соседних репозиториев и локальных окружений других сервисов;
- сохранить infrastructure- и межсервисные тесты доступными в полном локальном pipeline;
- задать единый, явный и расширяемый способ классификации infrastructure-тестов;
- сделать этот способ обязательным не только для существующих тестов, но и для каждого нового теста и будущего backend-микросервиса;
- гарантировать, что GitHub Actions продолжает проверять все автономные unit/API/in-process integration-тесты и не получает ложноположительный зелёный результат из-за случайного молчаливого skip;
- сохранить обязательные `make format`, `make lint`, `make test` для всех реализованных проектов.

**Non-Goals:**

- запуск PostgreSQL, Redis, соседних HAIntly-сервисов или полного compose-стека внутри GitHub Actions;
- изменение продуктового кода, API, БД, миграций, авторизации, фоновой обработки или событий;
- создание тестового pipeline отсутствующего `notification-service`;
- объединение сервисных репозиториев или изменение checkout/deploy topology.

## Decisions

### 1. `make test` является автономным CI-контрактом

Во всех реализованных проектах `make test` запускает максимально полный набор тестов, который работает после установки зависимостей текущего проекта. Команда не определяет GitHub Actions по переменным `CI`/`GITHUB_ACTIONS`: одинаковый вызов должен выбирать одинаковый набор локально и в CI.

Альтернатива — условно пропускать тесты только в GitHub Actions — отклонена, потому что локальный и CI pipeline становятся разными и проблема не воспроизводится обычной командой разработчика.

### 2. Infrastructure-тесты отмечаются marker `infrastructure`

Pytest-тест получает marker `infrastructure`, если ему требуется хотя бы одно из следующего:

- реальный PostgreSQL, Redis, NATS, Celery broker/backend или иной внешний процесс;
- Docker/Compose;
- миграции против реальной БД;
- запуск или checkout другого HAIntly-сервиса;
- секреты или доступ к внешнему провайдеру.

`make test` запускает pytest с исключением этого marker. Marker регистрируется в `pyproject.toml`, а неизвестные markers считаются ошибкой через strict markers. In-process HTTP-сервер, fake/mock адаптер и SQLite остаются автономными, если не требуют перечисленной инфраструктуры.

Альтернатива — исключать целые каталоги `tests/integration` — отклонена: в `main-be` уже есть автономные integration-тесты с локальным aiohttp-сервером, которые должны оставаться в CI.

### 3. Полный локальный набор запускается через `make test-infra`

Каждый backend-проект предоставляет `make test-infra`. При наличии infrastructure-тестов команда:

1. проверяет необходимые локальные зависимости и выдаёт конкретную инструкцию при их отсутствии;
2. загружает локальные test credentials и адреса только из окружения/`.docker-compose/.env`;
3. запускает именно тесты с marker `infrastructure`.

Для проектов без таких тестов цель остаётся частью обязательного backend-интерфейса и успешно сообщает, что применимых infrastructure-тестов нет. Полная локальная проверка затронутого backend-сервиса состоит из `make test` и `make test-infra`.

Альтернатива — оставить `make test` полным infrastructure-набором и добавить `make test-ci` — отклонена, потому что существующие GitHub Actions и базовый quality contract уже используют `make test`.

### 4. Межсервисный smoke `main-be` не предполагает структуру отдельного checkout

`test_profile_service_smoke.py` остаётся infrastructure-тестом. Его путь к `profile-service` и интерпретатор/команда запуска передаются через проверяемое окружение либо вычисляются только в HAIntly workspace с явной проверкой существования. При отсутствии зависимости `make test-infra` завершается понятной диагностикой; обычный `make test` этот smoke не собирает в исполняемый набор.

Smoke продолжает проверять существующий внутренний HTTP-контракт `main-be` → `profile-service`. Источник `X-User-Id`, service credential и входящая авторизация не меняются; тест не вводит новый endpoint или runtime hardcode.

### 5. `vacancy-service` разделяет автономные и infrastructure-проверки

PostgreSQL repository, Redis lock, реальные Alembic migration и зависящие от БД seeding-тесты помечаются `infrastructure`. API/unit-тесты с подменёнными repository/client зависимостями остаются в `make test`. Текущий `scripts/test.sh` перестаёт блокировать автономный набор проверкой Docker и используется/разделяется так, чтобы infrastructure preflight выполнялся только для `make test-infra`.

### 6. Остальные проекты проверяются на соответствие, но не получают искусственных infrastructure-тестов

`profile-service`, `ai-service` и `fastapi_template` должны пройти аудит и получить единый backend-интерфейс, включая `make test-infra`, но без искусственных infrastructure-тестов. `main-fe` проверяется только на автономность существующего `make test`: frontend не получает pytest marker или отдельную infrastructure-цель без реальной потребности.

### 7. `fastapi_template` является источником контракта для будущих backend-сервисов

Шаблон содержит зарегистрированный strict marker `infrastructure`, автономный `make test` и готовую цель `make test-infra`, даже если в самом шаблоне нет ни одного infrastructure-теста. При копировании шаблона новый backend-сервис получает одинаковый интерфейс без дополнительного архитектурного решения.

Каждый новый pytest-тест классифицируется разработчиком в момент создания по фактическим зависимостям. Автономный тест остаётся в `make test`; тест с внешней runtime-зависимостью получает marker `infrastructure` и запускается через `make test-infra`. Классификация не откладывается до первого падения CI.

Альтернатива — добавлять marker и цель только после появления первого infrastructure-теста — отклонена: это создаёт неоднородность новых сервисов и позволяет тестам случайно попасть в неправильный pipeline.

### 8. Workflow-документы обеспечивают долговременное соблюдение

`AGENTS.md` закрепляет контракт в протоколе нового backend-сервиса. `agents/backend.md` требует классифицировать каждый добавляемый или изменяемый тест и выполнять оба применимых набора. `agents/quality_gate.md` требует независимо проверить marker, отсутствие внешних зависимостей в `make test` и полноту `make test-infra`.

Обновление этих документов входит в scope как ремонт оркестратора по bootstrap-исключению: оно не меняет продуктовые границы, но сохраняет нормативное правило после синхронизации и архивирования change.

Альтернатива — оставить правило только в архивном OpenSpec change — отклонена, потому что будущий исполнитель может не получить этот change в контексте новой задачи.

### 9. Неизменяемые архитектурные области

- **API и HTTP:** новые endpoint и изменения существующих контрактов отсутствуют.
- **SSE и NATS JetStream:** события, subjects и доставка не меняются.
- **БД и миграции:** production-схемы и миграции не меняются; меняется только выбор уже существующих migration-тестов.
- **Фоновые процессы:** Celery-задачи и broker/backend не меняются.
- **Безопасность и персональные данные:** новые credentials не создаются; локальные секреты не выводятся в CI или отчёты.
- **HeadHunter и AI/LLM:** вызовы и адаптеры не меняются, поэтому сверка контрактов провайдеров для реализации не требуется.

## Risks / Trade-offs

- [Infrastructure-тест случайно останется без marker и снова сломает CI] → Добавить strict marker, аудит всего дерева тестов и Quality Gate с запуском `make test` в окружении без инфраструктуры.
- [Новый тест или сервис обойдёт контракт после завершения change] → Встроить интерфейс в `fastapi_template`, обязанности — в профили Backend Agent и Quality Gate, а правило создания сервиса — в `AGENTS.md`.
- [Разработчики будут запускать только быстрый набор] → Документировать `make test-infra` и обязать Backend Agent/Quality Gate выполнять обе команды для затронутых сервисов при доступной локальной инфраструктуре.
- [Слишком широкая маркировка уменьшит CI-покрытие] → Маркировать отдельные тесты/модули по фактической внешней зависимости, не исключать каталог `integration` целиком.
- [Межсервисный smoke зависит от layout workspace] → Сделать зависимость явной и валидируемой, с диагностикой до запуска subprocess.
- [Отсутствующая инфраструктура блокирует полную локальную проверку] → Автономный pipeline остаётся доступным; infrastructure-команда завершается с инструкцией запуска нужной инфраструктуры, а не молчаливым skip.

## Migration Plan

1. Зарегистрировать marker и разделить команды сначала в `main-be` и `vacancy-service`.
2. Классифицировать существующие тесты по фактическим зависимостям.
3. Проверить автономный pipeline без запущенных HAIntly-контейнеров и соседнего checkout.
4. Поднять локальную инфраструктуру и выполнить оба набора затронутых backend-сервисов.
5. Обновить `fastapi_template` и workflow-документы как нормативную основу для будущих сервисов и тестов.
6. Проверить неизменённый вызов `make test` в workflow всех реализованных проектов.

Rollback выполняется возвратом Makefile, pytest-конфигурации и markers; миграция данных и координация deployment не требуются.

## Open Questions

Нет. Имена `infrastructure` и `test-infra`, граница автономного набора и обязанность классифицировать каждый новый backend-тест фиксируются этим change.
