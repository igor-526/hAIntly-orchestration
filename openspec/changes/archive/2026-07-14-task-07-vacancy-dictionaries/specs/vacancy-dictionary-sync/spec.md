## ADDED Requirements

### Requirement: Нормализованное владение справочниками
`vacancy-service` MUST быть единственным владельцем локальных данных справочников HH и MUST хранить элементы `/dictionaries`, регионов, стран, категорий и профессиональных ролей, отраслей, городов/линий/станций метро и языков как типизированные реляционные записи, а не как JSON-дампы. Каждая запись MUST сохранять HH ID, изменяемые отображаемые поля, иерархическую связь где применимо и признак `active`.

#### Scenario: Сохранение иерархического справочника
- **WHEN** HH возвращает полный снимок регионов, отраслей, профессиональных ролей или метро
- **THEN** `vacancy-service` сохраняет каждый узел отдельной записью с его HH ID и связью с родителем или владельцем

#### Scenario: Другой сервис запрашивает данные
- **WHEN** `main-be` читает справочник
- **THEN** он использует HTTP API `vacancy-service` и не обращается к его PostgreSQL напрямую

### Requirement: Реестр состояния синхронизации
Система MUST вести отдельное состояние для `dictionaries`, `areas`, `countries`, `professional_roles`, `industries`, `metro` и `languages`, включая время последней попытки, время последней успешной полной синхронизации, текущий результат и безопасное описание ошибки.

#### Scenario: Успешная синхронизация
- **WHEN** снимок справочника полностью применён и DB-транзакция зафиксирована
- **THEN** состояние этого справочника отмечается успешным и `last_success_at` получает время завершения

#### Scenario: Ошибка до commit
- **WHEN** загрузка, валидация или применение снимка завершается ошибкой
- **THEN** ошибка фиксируется для этого справочника, а прежний `last_success_at` не изменяется

### Requirement: Планирование по отсутствию и возрасту
Celery Beat `vacancy-service` MUST каждый час ставить задачу проверки актуальности. Проверка MUST ставить отдельную задачу синхронизации для каждого отсутствующего, никогда успешно не загруженного или просроченного справочника и MUST ничего не ставить для актуальных справочников. Максимальный возраст в часах MUST поступать из обязательной положительной настройки окружения и валидироваться при старте.

#### Scenario: Все справочники отсутствуют
- **WHEN** часовая проверка не находит ни одной записи успешной синхронизации
- **THEN** она ставит полную синхронизацию каждого из семи логических справочников

#### Scenario: Отсутствуют некоторые справочники
- **WHEN** часть справочников имеет актуальный успешный снимок, а часть отсутствует
- **THEN** задачи создаются только для отсутствующих справочников

#### Scenario: Некоторые справочники просрочены
- **WHEN** время после `last_success_at` части справочников равно или превышает настроенное количество часов
- **THEN** задачи создаются только для просроченных справочников

#### Scenario: Все справочники актуальны
- **WHEN** каждый справочник имеет успешный снимок моложе настроенного порога
- **THEN** проверка не создаёт задач синхронизации

#### Scenario: Некорректный период
- **WHEN** период отсутствует, равен нулю, отрицателен или не является целым числом
- **THEN** `vacancy-service`, worker и Beat завершают загрузку настроек конфигурационной ошибкой до обслуживания задач

### Requirement: Идемпотентное применение полного снимка
Синхронизация каждого логического справочника MUST в одной DB-транзакции выполнять upsert по HH ID в пространстве этого справочника, обновлять изменившиеся поля, устанавливать `active=true` присутствующим элементам и `active=false` ранее известным элементам, отсутствующим в новом полном снимке. Повтор того же снимка MUST приводить к тому же состоянию без дубликатов.

#### Scenario: HH изменил название
- **WHEN** полный снимок содержит существующий HH ID с новым названием
- **THEN** запись обновляется без создания нового элемента и остаётся активной

#### Scenario: Элемент исчез из снимка
- **WHEN** ранее активный HH ID отсутствует в валидном полном снимке своего справочника
- **THEN** запись сохраняется, но получает `active=false`

#### Scenario: Элемент вернулся
- **WHEN** ранее неактивный HH ID снова присутствует в полном снимке
- **THEN** его поля обновляются и `active` становится `true`

#### Scenario: Повторная доставка задачи
- **WHEN** worker повторно применяет уже успешно применённый снимок
- **THEN** уникальные записи, связи и активность не изменяются семантически и дубликаты не появляются

### Requirement: Защита последнего успешного снимка
Пустой вопреки контракту, неполный или структурно невалидный ответ HH MUST NOT применяться как новый полный снимок. Ошибка одного справочника MUST NOT откатывать уже успешно синхронизированные независимые справочники.

#### Scenario: Неожиданно пустой ответ
- **WHEN** HH возвращает для непустого по контракту справочника пустой или невалидный payload
- **THEN** транзакция этого снимка откатывается, существующая активность и `last_success_at` сохраняются

#### Scenario: Частичный отказ
- **WHEN** один из семи справочников завершается ошибкой, а другой успешно применён
- **THEN** успешный справочник фиксируется независимо, а ошибочный сохраняет предыдущий снимок

### Requirement: Безопасная интеграция с HH
HH-адаптер MUST использовать подтверждённые OpenAPI GET-операции `/dictionaries`, `/areas`, `/areas/{area_id}`, `/areas/countries`, `/professional_roles`, `/industries`, `/metro`, `/metro/{city_id}` и `/languages`, MUST отправлять настроенный `HH-User-Agent`, ограничивать timeout/retry и MUST получать базовый URL и связанные параметры из валидируемого окружения без production hardcode. Пользовательский HH OAuth-токен MUST NOT требоваться или сохраняться для этих общедоступных справочников.

#### Scenario: Успешное получение общедоступного справочника
- **WHEN** worker запрашивает справочник и HH отвечает валидным `200`
- **THEN** адаптер валидирует типизированный payload и передаёт его слою синхронизации без пользовательского токена

#### Scenario: Временная ошибка HH
- **WHEN** HH отвечает временной сетевой ошибкой, `429` или `5xx`
- **THEN** адаптер выполняет только настроенное ограниченное число повторов с backoff, после чего задача завершается диагностируемой ошибкой без изменения успешного снимка

### Requirement: Исключение конкурентной синхронизации
В один момент времени MUST выполняться не более одной задачи применения для одного логического справочника; блокировка MUST использовать инфраструктурную границу Redis и иметь ограниченный срок жизни, а DB MUST дополнительно обеспечивать уникальность доменных ключей.

#### Scenario: Две задачи одного справочника
- **WHEN** повторная доставка совпадает с уже выполняющейся синхронизацией того же справочника
- **THEN** только один worker получает lock и применяет снимок, а второй безопасно завершается или повторяется позднее

### Requirement: Наблюдаемость без утечки данных
Фоновая обработка MUST логировать имя справочника, результат, длительность и агрегированные счётчики изменений и MUST NOT логировать credentials, Redis URL, заголовки авторизации или полные payload HH.

#### Scenario: Синхронизация завершена ошибкой
- **WHEN** задача не может получить или применить снимок
- **THEN** логи позволяют определить справочник и класс ошибки без раскрытия секретов или полного внешнего ответа
