## Why

Для поиска вакансий и будущей формы фильтров HAIntly нужны локальные, быстро читаемые и регулярно обновляемые справочники HeadHunter. Сейчас `vacancy-service` не владеет таким набором данных, поэтому зависимость от изменяемых значений HH нельзя надёжно использовать в пользовательских сценариях.

## What Changes

- `vacancy-service` получает нормализованное хранение общедоступных справочников HH: справочников полей, регионов и стран, профессиональных ролей, отраслей, метро и языков.
- Добавляется реестр состояния синхронизации по каждому логическому справочнику, включая время последней успешной синхронизации и результат последней попытки.
- Добавляется идемпотентная синхронизация через HH API: upsert по идентификатору HH, обновление изменившихся полей и деактивация отсутствующих в очередном полном снимке элементов.
- Celery Beat каждый час проверяет актуальность и запускает синхронизацию отсутствующих или просроченных справочников; допустимый возраст задаётся окружением в часах.
- `vacancy-service` предоставляет внутренние read-only LIST/GET endpoints для активных элементов справочников с регистронезависимым текстовым фильтром; `main-be` публикует соответствующий авторизованный proxy API для `main-fe`.
- Добавляются миграции, конфигурация Redis/Celery, отдельные процессы worker/beat в compose, модульные, интеграционные и e2e-проверки фоновой синхронизации и API.
- В scope не входят UI формы фильтров, поиск вакансий по этим значениям, ручное изменение справочников, публичные write endpoints, пользовательская синхронизация по запросу и хранение персональных данных или HH OAuth-токенов.

## Capabilities

### New Capabilities

- `vacancy-dictionary-sync`: хранение, планирование и безопасная синхронизация общедоступных справочников HeadHunter в БД `vacancy-service`.
- `vacancy-dictionary-api`: read-only LIST/GET доступ к активным значениям справочников через внутренний API `vacancy-service` и авторизованный proxy `main-be`.

### Modified Capabilities

- `local-compose-runtime`: локальный runtime `vacancy-service` дополняется собственными Celery worker и Celery Beat, использующими общий инфраструктурный Redis без добавления Redis или PostgreSQL в compose сервиса.

## Impact

- `vacancy-service`: новые модели и миграции PostgreSQL, HH-клиент/адаптер, Celery-задачи и расписание, конфигурация, read-only HTTP API и тесты. Сервис остаётся владельцем всех данных справочников.
- `main-be`: новые публичные авторизованные proxy endpoints и конфигурация адреса/сервисного credential для обращения к `vacancy-service`; собственное хранение справочников не добавляется.
- `.docker-compose/docker-compose.vacancy.yml`, корневой и сервисный Makefile, безопасные env-примеры: процессы приложения, миграций, worker и beat используют общую инфраструктуру и валидируемые настройки окружения.
- HeadHunter API: используются общедоступные GET `/dictionaries`, `/areas`, `/areas/{area_id}`, `/areas/countries`, `/professional_roles`, `/industries`, `/metro`, `/metro/{city_id}` и `/languages`; входящие пользовательские токены не требуются.
- `main-fe`, NATS JetStream, SSE, другие сервисы и их БД не изменяются.
