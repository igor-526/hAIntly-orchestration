## Context

`vacancy-service` уже выделен как владелец данных вакансий, но пока является архитектурной заготовкой и не хранит общедоступные справочники HH. Эти справочники нужны будущим фильтрам, при этом HH прямо указывает, что их значения могут измениться. Локальная `docs/hh_openapi.yml` и актуальный официальный Redoc подтверждают GET-контракты `/dictionaries`, `/areas`, `/areas/{area_id}`, `/areas/countries`, `/professional_roles`, `/industries`, `/metro`, `/metro/{city_id}` и `/languages`, включая древовидные ответы регионов, отраслей, ролей и метро.

Владелец всех локальных записей и состояния их синхронизации — `vacancy-service`. Инициатор плановой проверки — Celery Beat этого сервиса, исполнитель — его Celery worker, внешний получатель запросов — HH API по HTTPS. Пользовательский поток чтения инициирует `main-fe`, но проходит по HTTP через авторизованный `main-be` к внутреннему HTTP API `vacancy-service`; прямого обращения frontend к профильному сервису нет.

## Goals / Non-Goals

**Goals:**

- хранить каждый элемент справочника как типизированную реляционную запись, сохраняя иерархию, HH ID, отображаемые поля и `active`;
- атомарно и идемпотентно обновлять полный снимок каждого логического справочника и деактивировать исчезнувшие элементы;
- проверять актуальность каждый час, а допустимый возраст успешного снимка задавать окружением в часах;
- предоставить стабильный read-only LIST/GET API активных записей с пагинацией и регистронезависимым текстовым поиском;
- обеспечить доступ к API через `main-be`, сохранив сервисные границы и пользовательскую аутентификацию на публичной точке входа;
- обеспечить наблюдаемость, повторяемые тесты и e2e четырёх заданных состояний актуальности.

**Non-Goals:**

- UI и подстановка значений в форму фильтров;
- поиск вакансий, пользовательские статусы вакансий и интеграция с AI/NATS/SSE;
- write API, ручной запуск или редактирование справочников пользователем;
- хранение сырых JSON-дампов, HH OAuth-токенов или персональных данных;
- синхронизация иных HH-справочников.

## Decisions

### 1. Реляционная модель следует структуре справочников

Миграции `vacancy-service` создают отдельные таблицы для: элементов `/dictionaries` с обязательным кодом набора; регионов; стран; категорий и профессиональных ролей; отраслей; городов, линий и станций метро; языков; а также реестр синхронизаций. Иерархические сущности хранят ссылку на родителя или таблицу-владельца. Каждая предметная запись содержит стабильный HH ID, изменяемые отображаемые поля, `active` и timestamps; уникальные ограничения соответствуют пространству идентификаторов конкретного справочника. JSON-колонки и межсервисный доступ к БД не используются.

Альтернатива — одна универсальная key/value-таблица — отвергнута: она ослабляет внешние ключи, типизацию, индексы и проверяемость иерархий. Отдельная таблица на каждый ключ внутри ответа `/dictionaries` также отвергнута, потому что набор ключей управляется HH; код набора является частью составного доменного ключа.

Реестр синхронизации имеет заранее определённую запись на каждый логический снимок: `dictionaries`, `areas`, `countries`, `professional_roles`, `industries`, `metro`, `languages`. Он хранит как минимум состояние, время начала/окончания последней попытки, время последнего успешного полного снимка и безопасное описание последней ошибки.

### 2. Полный снимок применяется транзакционно

HH-адаптер изолирует aiohttp и типизированно реализует перечисленные в OpenAPI GET-операции, обязательный `HH-User-Agent`, таймауты и ограниченные повторы временных сетевых/5xx/429 ошибок. Базовый URL, user-agent, таймауты и retry-параметры валидируются из окружения; production hardcode запрещён. Общедоступные справочники не требуют пользовательского OAuth-токена.

Для каждого логического справочника worker получает полный снимок, разворачивает дерево в записи, затем в одной DB-транзакции выполняет upsert по HH ID (и коду набора там, где ID локален набору), активирует присутствующие записи и деактивирует ранее активные записи, отсутствующие в снимке. `last_success_at` обновляется только после commit. Пустой или структурно невалидный ответ не считается доказательством удаления всех значений: транзакция откатывается, активность и `last_success_at` сохраняются, попытка отмечается ошибкой.

Снимки независимы: ошибка одного не откатывает успешно завершённые другие. Для одного логического снимка используется распределённая блокировка/идемпотентный lock на общем Redis, чтобы Beat и повторная доставка не создавали конкурентные записи. Альтернатива «truncate и insert» отвергнута, поскольку ломает стабильные ссылки и скрывает исчезнувшие значения вместо сохранения их как неактивных.

### 3. Beat проверяет, worker синхронизирует

Celery Beat `vacancy-service` каждый час ставит одну задачу проверки. Проверка читает реестр и ставит отдельную задачу только для отсутствующего, никогда успешно не синхронизированного либо просроченного снимка. Просрочка определяется как `now - last_success_at >= VACANCY_DICTIONARY_MAX_AGE_HOURS`; значение обязательно положительное и валидируется при старте. Проверка не ждёт сетевых операций.

Celery broker/backend и lock используют общий инфраструктурный Redis через отдельные обязательные URL/настройки окружения `vacancy-service`. Worker и Beat являются отдельными процессами этого микросервиса в `docker-compose.vacancy.yml`; compose не создаёт собственный Redis или PostgreSQL. Логи содержат имя справочника, попытку, длительность и счётчики insert/update/deactivate без payload, credentials или заголовков.

### 4. API является read-only и отделяет публичную границу от внутренней

`vacancy-service` предоставляет ресурсные LIST/GET endpoints под внутренним namespace справочников. Конкретные пути и схемы фиксируются реализацией согласованно для семейств `dictionaries`, `areas`, `countries`, `professional-roles`, `industries`, `metro-cities`, `metro-lines`, `metro-stations`, `languages`; LIST поддерживает ограниченные `limit`/`offset`, `q` и необходимые родительские фильтры, GET принимает локально однозначный HH ID. По умолчанию выдаются только `active=true`; неактивный или неизвестный ID возвращает `404`. `q` сопоставляется с отображаемым именем регистронезависимо как буквальный текст, экранированный перед применением регулярного выражения, чтобы ввод пользователя не менял смысл или стоимость шаблона.

`main-be` предоставляет зеркальный публичный read-only namespace и проксирует запрос после существующей cookie-auth. Адрес `vacancy-service`, timeout и исходящий service credential, если единый межсервисный middleware проекта требует его, поступают только из валидируемого окружения; в рамках этого change новая входящая авторизация у закрытого `vacancy-service` не вводится. `main-be` передаёт UUID аутентифицированного пользователя в `X-User-Id`; `vacancy-service` валидирует формат UUID для трассировки, не использует его для фильтрации общих данных и отвечает `400` при отсутствующем/невалидном заголовке. Публичный API без действующей cookie отвечает существующим `401`; внешние ошибки профильного сервиса преобразуются в стабильный gateway-ответ без утечки внутренних деталей.

Альтернатива — отдавать справочники из `main-be` DB — нарушает владение данными. Прямой доступ `main-fe` к `vacancy-service` нарушает утверждённую gateway-границу.

### 5. События, безопасность и персональные данные

Change не добавляет NATS JetStream subjects, SSE или межсервисные события: фоновые задания внутренние и используют Celery. Справочники общедоступны и не содержат персональных данных. `X-User-Id` используется только как контекст запроса и не сохраняется в таблицах справочников. Ответы HH, service credentials, Redis URL и пользовательские идентификаторы не логируются целиком. POST/PATCH/DELETE для справочников отсутствуют.

## Risks / Trade-offs

- [Изменение структуры ответа HH] → строгая валидация адаптера, откат только проблемного снимка, сохранение последнего успешного набора и диагностируемая ошибка.
- [Пустой ошибочный ответ деактивирует всё] → запрет применения неожиданно пустого/невалидного снимка и транзакционная проверка до деактивации.
- [Параллельные Celery-задачи] → lock на логический справочник, уникальные ограничения и идемпотентный upsert.
- [Большой объём и медленный regex] → обязательная пагинация, индексы на нормализованные имена, экранирование `q`, лимит длины и времени запроса.
- [HH rate limiting или недоступность] → таймаут, bounded retry с backoff, отсутствие обновления `last_success_at`; текущие активные данные продолжают читаться.
- [Частично свежий каталог] → состояние и транзакция ведутся отдельно по семействам; API остаётся доступным с последним успешным снимком.

## Migration Plan

1. Добавить зависимости и валидируемую конфигурацию, миграции таблиц и индексов.
2. Развернуть `vacancy-service` app/migration/worker/beat с тем же compose project `haintly-vacancy-service`, общей сетью и инфраструктурными PostgreSQL/Redis.
3. После миграции Beat обнаружит отсутствующие записи реестра и поставит первичную синхронизацию; готовность HTTP приложения не зависит от доступности HH, но состояние синхронизации наблюдаемо в логах/БД.
4. Развернуть proxy API `main-be` после доступности внутреннего API и выполнить direct/proxy smoke.
5. При rollback остановить Beat/worker и откатить приложение/API; таблицы не удалять до подтверждения, что новые данные не нужны. Повторное развёртывание безопасно продолжает upsert.
