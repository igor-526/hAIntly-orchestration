## Context

`services/main-fe` — Next.js-приложение, которое взаимодействует только с публичной границей `main-be` по HTTP и SSE. Сейчас часть исходников находится вне единого `src`, routing, UI, состояние и сетевое взаимодействие недостаточно разделены, базовая ESLint-конфигурация не задаёт нужную строгость, а `make fe-build` не является подтверждённым воспроизводимым Quality Gate.

В качестве архитектурного эталона используется `/home/igor/projects/eqSiteCMS/services/frontend`: единый `src`, feature-модули с локальными hooks/services/ui/validators, общие слои и тестовая инфраструктура. `/home/igor/projects/nexora-seo/orchestration/services/fe` используется как дополнительный референс строгих ESLint-правил. Копирование доменной логики, зависимостей или исключений из референсов не требуется.

Изменение затрагивает весь frontend и bootstrap-инструкции двух профильных агентов, но не меняет пользовательские сценарии или межсервисные контракты. Полный перенос должен выполняться как одна согласованная миграция с регрессионными проверками.

## Goals / Non-Goals

**Goals:**

- Сделать `src` единственным корнем исходного кода приложения и закрепить однозначные ответственности каталогов.
- Разделить routing, feature orchestration, сетевую границу, типы и переиспользуемый UI без циклических или обратных зависимостей.
- Сохранить существующие URL, UX, HTTP/SSE-запросы и пользовательские состояния.
- Сделать lint, test, typecheck, image build и runtime обязательными воспроизводимыми проверками.
- Гарантировать, что `make fe-build` выполняется в контейнерном build context и не создаёт/не изменяет `.next`, `out`, `build`, coverage или TypeScript build-info в рабочем дереве.
- Зафиксировать архитектуру и проверки в профилях Frontend и Quality Gate Agent.

**Non-Goals:**

- Редизайн, новые пользовательские возможности или изменение существующих маршрутов.
- Изменение публичного API/SSE-контракта `main-be`, backend-кода или сервисных границ.
- Замена Next.js, React, UI-библиотеки или package manager без отдельной необходимости, доказанной реализацией.
- Внедрение API code generation, Storybook или иных инструментов только потому, что они присутствуют в референсах.

## Decisions

### 1. Единый корень и направленные зависимости

Все runtime-исходники, стили, тестовые utilities и типы приложения размещаются под `services/main-fe/src`:

```text
src/
├── api/       # типизированные HTTP/SSE-клиенты только для main-be
├── app/       # Next.js routing, layouts и route metadata
├── contexts/  # общие React contexts/providers
├── features/  # пользовательские сценарии по отдельным feature
├── hooks/     # общие React hooks
├── lib/       # framework-neutral общие функции и конфигурация
├── test/      # setup, render helpers, fixtures и transport mocks
├── types/     # общие типы приложения и API
└── ui/        # UI, реально переиспользуемый несколькими feature
```

Статические assets остаются в `public`, а build/tooling-конфигурация — в корне `services/main-fe`. Алиас `@/*` указывает на `./src/*`; Vitest использует тот же alias.

Разрешённая цепочка: `app -> feature ui -> feature hook -> feature service -> api -> main-be`. Feature может использовать общие `contexts`, `hooks`, `lib`, `types`, `ui`; shared-слои не импортируют feature или `app`, а feature не импортируют внутренности другой feature. Каждый route page экспортирует один feature page-компонент и не содержит запросов, состояния сценария либо бизнес-оркестрации. В `features/<name>` создаются применимые `hooks`, `services`, `ui`, `validators`; пустые каталоги ради формы не создаются.

Альтернатива — сохранить текущие пути и постепенно переносить новые файлы — отклонена: она оставляет две архитектуры и не выполняет требование полного рефакторинга.

### 2. Сетевая граница и сохранение поведения

Владелец пользовательских и доменных данных не меняется. Пользователь инициирует действие в feature UI; feature service вызывает типизированный клиент из `src/api`; получатель — только `main-be`, протокол — существующий HTTP. Для уведомлений инициатором соединения является `main-fe`, получателем подключения — `main-be`, протокол — существующий SSE; события передаются hook/context соответствующей feature без изменения wire-формата.

Прямые обращения к профильным сервисам, HH, AI-провайдеру, NATS или БД запрещены. Адрес `main-be` берётся из существующего валидируемого frontend environment, без hardcode. Cookie/user context продолжает обрабатываться существующим публичным контрактом; внутренний `X-User-Id` и service credential к frontend не относятся и не добавляются.

API endpoints, SSE events, базы данных, фоновые процессы и персональные данные этим change не изменяются. Тестовые fixtures не должны содержать реальные credentials или персональные данные; runtime-код не должен добавлять логирование чувствительных значений.

### 3. Feature-локальность и критерий shared-кода

Компонент-страница принадлежит `features/<name>/ui`; специфичные состоянию React hooks, orchestration services и validators находятся рядом с feature. Код переносится в корневые `hooks`, `lib`, `types` или `ui` только при фактическом использовании более чем одной feature либо когда он образует общую инфраструктурную границу. Это предотвращает преждевременный shared-слой и скрытые связи.

Альтернатива — группировать файлы только по техническому типу на уровне всего приложения — отклонена, поскольку она размывает владельца пользовательского сценария.

### 4. Строгий lint без подавления

Текущие Next.js Core Web Vitals и TypeScript presets сохраняются. К ним добавляются совместимые строгие правила из референсов: запрет `any`, неиспользуемого и дублирующего кода/imports, опасных TypeScript comments, неуместного console, чрезмерной сложности и крупных модулей/функций, а также сложной inline-логики обработчиков. Конкретный набор и уровни фиксируются конфигурацией и lock-файлом; существующее правило нельзя понизить с error до warning/off, а новые нарушения нельзя обходить eslint-disable, расширением ignore или исключением файлов. Допустимые тестовые overrides должны быть узкими, мотивированными особенностями тестов и не отключать архитектурные ограничения.

Альтернатива — сначала ослабить lint и ужесточать позднее — отклонена как противоречащая цели change.

### 5. Тестовая стратегия

Vitest/jsdom и Testing Library остаются базой. `src/test` централизует setup, custom render/providers, fixtures и mock HTTP/SSE boundary. Регрессионные тесты подтверждают прежние маршруты и критические состояния существующих feature; hooks/services проверяют success, edge/empty и error, компоненты — loading/empty/error/data и interactions. Тесты не требуют live backend.

Минимальный локальный gate: `npm test`, `npm run lint`, `npx tsc --noEmit`. Проверки запускаются из `services/main-fe` на чистом dependency install, а package scripts/config включают весь `src`. Typecheck настраивается так, чтобы не создавать или не изменять build-info в рабочем дереве.

### 6. Контейнерная сборка, runtime и чистота дерева

Корневая `make fe-build` остаётся канонической командой сборки образа и использует `.docker-compose/docker-compose.fe.yml`. Docker build context, Dockerfile и `.dockerignore` включают необходимые manifest/config/src/public файлы и исключают host build/test artifacts. Сборка выполняет production `next build` внутри image stage; артефакты не bind-mount и не копируются обратно на host.

После сборки Frontend Agent запускает `make fe`, проверяет опубликованный frontend-порт, состояние и логи контейнера. До и после `make fe-build` и всех проверок сравнивается рабочее дерево; `.next`, `out`, `build`, `coverage`, временные cache/build-info не должны оставаться новыми или изменёнными. Существующий tracked build-info удаляется из репозитория и игнорируется, если не является необходимым source artifact.

### 7. Workflow как часть bootstrap-ремонта

`agents/frontend.md` получает нормативную карту `src`, правила зависимостей, критерий shared-кода, запрет lint-ослаблений и точные команды передачи. `agents/quality_gate.md` независимо повторяет `npm test`, `npm run lint`, `npx tsc --noEmit`, `make fe-build`, runtime/port/log checks и проверку чистоты артефактов. Reviewer возвращает `НА ДОРАБОТКУ` при пропуске или ошибке любой обязательной проверки и создаёт отчёт только после полного успеха.

## Risks / Trade-offs

- [Большой перенос создаёт merge-конфликты и риск пропущенных import] → выполнять перенос по feature, использовать alias только после переноса, затем проверить отсутствие runtime-файлов вне `src` и прогнать полный gate.
- [Строгие правила дадут много нарушений одновременно] → исправлять код, декомпозировать функции и удалять dead code; не понижать severity и не расширять ignores.
- [Тесты могут зафиксировать детали реализации] → проверять пользовательское поведение и публичные module boundaries, использовать единый render helper и transport mocks.
- [Next.js генерирует `.next` и type metadata] → production build выполнять внутри Docker; локальные команды конфигурировать без persistent build-info, generated пути игнорировать и проверять git status до/после.
- [Механический перенос может незаметно изменить HTTP/SSE semantics] → сохранить request/response mapping, добавить boundary/regression tests и выполнить runtime smoke через опубликованный frontend.
- [Ужесточение lint увеличивает стоимость будущих изменений] → это сознательный trade-off ради раннего выявления архитектурных и поддерживаемостных дефектов.

## Migration Plan

1. Зафиксировать baseline маршрутов, feature, API/SSE-вызовов, тестов и текущей причины падения `make fe-build`.
2. Подготовить `src`, alias и тестовую инфраструктуру, затем последовательно перенести shared API/types/lib/UI и каждую feature с её тестами.
3. Свести route pages к одному feature page-компоненту и удалить дубли/старые исходники вне `src`.
4. Ужесточить ESLint, исправить все нарушения и обновить package/TypeScript/Vitest настройки и lock-файл.
5. Исправить Docker/compose/Makefile boundary, выполнить локальный gate, image build, runtime smoke и проверку чистоты дерева.
6. Обновить профили агентов и передать отдельному Quality Gate Agent.

Откат выполняется возвратом всего change одним согласованным набором: смешанное состояние старых и новых путей не поддерживается. Изменений данных или backend deployment нет.
