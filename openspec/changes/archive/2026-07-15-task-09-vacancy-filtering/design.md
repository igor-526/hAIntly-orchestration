## Context

В `vacancy-service` работают справочники HH (task-07) и пресеты фильтров (task-08). Internal API отдаёт CRUD для пресетов и read-only данные справочников. `main-be` проксирует всё с cookie-auth. `main-fe` имеет Drawer фильтров с формой, но кнопка «Применить» — заглушка: поиск вакансий не реализован.

Для поиска вакансий необходимо: (1) получить актуальный HH access token от `profile-service`, (2) передать параметры поиска в HH API `GET /vacancies`, (3) отобразить результаты в UI. Отдельная вакансия получается по `GET /vacancies/{id}`.

Сейчас `profile-service` хранит зашифрованные токены, но не предоставляет endpoint для их выдачи другим сервисам. Refresh token хранится, но механизм refresh не реализован.

## Goals / Non-Goals

**Goals:**

- Реализовать поиск вакансий через HH API с поддержкой всех не-deprecated query-параметров и пресетов фильтров.
- Реализовать получение деталей одной вакансии по ID.
- Предоставить сервисный endpoint в `profile-service` для выдачи расшифрованного access token с опциональным refresh.
- Proxy-доступ к поиску/просмотру вакансий через `main-be` с cookie-auth.
- Отображение результатов поиска в `main-fe` с layout «список слева, детали справа».
- Интеграция кнопки «Применить» в Drawer фильтров с фактическим поиском.

**Non-Goals:**

- Хранение вакансий в БД `vacancy-service` (вакансии запрашиваются в реальном времени из HH API).
- Пользовательские статусы вакансий, AI-оценка, сопроводительные письма (будущие итерации).
- Настройка отображения списка и деталей (будущие итерации).
- Кеширование результатов поиска.

## Decisions

### 1. Application token для поиска вакансий

**Решение:** HH API `GET /vacancies` и `GET /vacancies/{id}` требуют авторизации приложения (application token), а не пользователя. Vacancy-service использует `HH_APP_TOKEN` из окружения — бессрочный токен, не требующий refresh.

**Почему не user token:** HH API `/vacancies` работает с application-level авторизацией. User token нужен только для операций от имени пользователя (отклики, резюме).

**Альтернатива:** Получать user token через profile-service.  
**Почему нет:** Для поиска вакансий это избыточно; profile-service используется только для получения `hh_user_id` при работе с пресетами.

### 2. Profile-service для hh_user_id (пресеты)

**Решение:** При использовании пресетов фильтров main-be получает `hh_user_id` от profile-service и передаёт его в vacancy-service через `X-Hh-User-Id` заголовок. Vacancy-service использует `hh_user_id` для загрузки пресета из БД. Без пресета `X-Hh-User-Id` не требуется.

**Альтернатива:** Vacancy-service сам обращается к profile-service.  
**Почему нет:** Лишний HTTP-запрос; main-be уже имеет связь с profile-service.

### 3. Refresh token в profile-service (для будущего использования)

**Решение:** Endpoint `GET /internal/hh-token/{account_id}` с поддержкой refresh реализован в profile-service для будущего использования (отклики, резюме, другие операции от имени пользователя). Сейчас для поиска вакансий он не используется.

### 4. API endpoints в vacancy-service

| Метод | Path | Описание |
|---|---|---|
| `GET` | `/internal/vacancies` | Поиск вакансий (проксирует HH `GET /vacancies`) |
| `GET` | `/internal/vacancies/{vacancy_id}` | Детали вакансии (проксирует HH `GET /vacancies/{id}`) |

Endpoints требуют `X-User-Id` (UUID) для контекста пользователя. `X-Hh-User-Id` (строка) опционален — нужен только при использовании пресетов. Vacancy-service:
1. Использует `HH_APP_TOKEN` для авторизации в HH API.
2. При `preset_id` загружает пресет по `hh_user_id` из заголовка.
3. Вызывает HH API с application token.
4. Возвращает ответ клиенту.

#### Обработка пресетов

`GET /internal/vacancies` принимает:
- Все не-deprecated query-параметры HH `GET /vacancies` (как nullable поля).
- Дополнительный параметр `preset_id` (UUID, nullable).

Логика обработки (из ТЗ):
- **Все null (включая preset_id):** запрос к HH без фильтрации.
- **Поля не null, preset_id null:** запрос к HH с переданными параметрами.
- **Только preset_id:** загрузить пресет, использовать его параметры.
- **preset_id + поля:** загрузить пресет, переопределить поля переданными значениями (без сохранения).

Различие между явным `null` и отсутствием параметра: vacancy-service проверяет наличие ключа в query-параметрах. Если ключ присутствует со значением `null`/пустая строка — параметр считается переданным (для скалярных это значит «сбросить»). Если ключ отсутствует — параметр не передан.

### 5. Proxy endpoints в main-be

| Метод | Path | Описание |
|---|---|---|
| `GET` | `/api/vacancies` | Поиск вакансий (прокси к vacancy-service) |
| `GET` | `/api/vacancies/{vacancy_id}` | Детали вакансии (прокси к vacancy-service) |

main-be при каждом запросе:
1. Получает `user_id` из cookie-auth.
2. Передаёт `X-User-Id` в vacancy-service.
3. Если в параметрах есть `preset_id` — получает `hh_user_id` от profile-service и передаёт в `X-Hh-User-Id`.

### 6. Интеграция пресетов фильтров с поиском

**Решение:** При нажатии «Применить» в Drawer фильтров frontend:
1. Собирает текущие значения формы.
2. Если выбран пресет — передаёт `preset_id` + переопределённые поля.
3. Если пресет не выбран — передаёт только поля формы.
4. Вызывает `GET /api/vacancies` с параметрами.
5. Закрывает Drawer.
6. Отображает результаты в левой панели.

### 7. Фронтенд: layout и компоненты

**Решение:** В `WorkspaceShell`:
- Левая панель: `VacancyList` — список карточек вакансий (имя, компания, зарплата, локация, snippet).
- Правая панель: `VacancyDetail` — полная информация о выбранной вакансии.
- При первичной загрузке — получить список вакансий без фильтрации.
- При клике на карточку — загрузить и показать детали справа.
- Если вакансия не выбрана — текст-подсказка «Выберите вакансию из списка».

### 8. Хранение состояния поиска и пагинация

**Решение:** Хук `useVacancySearch` управляет:
- `vacancies` — текущий список результатов (накапливается при подгрузке).
- `selectedId` — ID выбранной вакансии.
- `vacancy` — детали выбранной вакансии.
- `loading` / `error` — состояния загрузки.
- `page` — текущая страница (начинается с 0).
- `hasMore` — есть ли ещё страницы для подгрузки.
- `search(params)` — запуск нового поиска (сбрасывает список и page на 0).
- `loadMore()` — подгрузка следующей страницы (инкрементирует page, дополняет vacancies).
- `select(id)` — выбор вакансии для просмотра деталей.

Состояние не сохраняется между перезагрузками страницы.

### 10. Infinite scroll пагинация

**Решение:** Пагинация реализована как infinite scroll с подгрузкой по 30 вакансий за раз (`per_page=30`). HH API поддерживает `page` (0-indexed) и `per_page` (max 100). Логика:
1. Новый поиск — `page=0`, `per_page=30`, список сбрасывается.
2. При скролле до конца левой панели — `loadMore()` запрашивает `page+1`.
3. Если вернулось меньше 30 вакансий или `page >= pages` — `hasMore=false`, подгрузка прекращается.
4. Максимальная глубина HH API — 2000 результатов (`per_page=10`, `page=199` последний). При `per_page=30` максимальная глубина — `page=66` (30×66=1980). При достижении лимита `hasMore=false`.

**Альтернатива:** Кнопка «Показать ещё».  
**Почему нет:** Infinite scroll привычнее для списков вакансий; не требует дополнительного действия пользователя.

**Реализация:** В `VacancyList` используется `IntersectionObserver` на sentinel-элементе внизу списка. Когда sentinel входит в viewport — вызывается `loadMore()`.

### 9. X-Hh-User-Id только для пресетов

**Решение:** `X-Hh-User-Id` передаётся из main-be в vacancy-service только при использовании пресетов (`preset_id` в параметрах). Для обычного поиска заголовок не требуется. Это минимизирует обращения к profile-service.

## Risks / Trade-offs

| Риск | Митигация |
|---|---|
| HH rate limiting при частых запросах поиска | Пользователь инициирует поиск вручную; автоматических polling нет. |
| Application token может быть отозван | Токен хранится в .env; при отзыве — обновить .env и перезапустить сервис. |
| HH API возвращает максимум 2000 результатов | Infinite scroll подгружает постранично; при достижении лимита `hasMore=false` и подгрузка прекращается. |
| Форма фильтров содержит 20+ полей — сложная сборка параметров | Переиспользуется логика пресетов из task-08; параметры маппятся 1:1 с HH query. |
