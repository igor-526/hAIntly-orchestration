## MODIFIED Requirements

### Requirement: Нормализованное владение справочниками
`vacancy-service` MUST быть единственным владельцем локальных данных справочников HH и MUST хранить элементы `/dictionaries`, регионов, стран, категорий и профессиональных ролей, отраслей, городов/линий/станций метро и языков как типизированные реляционные записи, а не как JSON-дампы. Каждая запись MUST иметь внутренний UUID primary key, MUST сохранять HH ID как отдельный domain key, изменяемые отображаемые поля, иерархическую связь где применимо и признак `active`. Уникальность и upsert MUST определяться по HH domain key в пространстве конкретного справочника, а не по внутреннему UUID.

#### Scenario: Сохранение иерархического справочника
- **WHEN** HH возвращает полный снимок регионов, отраслей, профессиональных ролей или метро
- **THEN** `vacancy-service` сохраняет каждый узел отдельной записью с UUID primary key, его HH ID и связью с родителем или владельцем

#### Scenario: Другой сервис запрашивает данные
- **WHEN** `main-be` читает справочник
- **THEN** он использует HTTP API `vacancy-service` по существующему HH ID и не обращается к его PostgreSQL или внутреннему UUID напрямую

#### Scenario: Повторное применение domain key
- **WHEN** синхронизация повторно получает уже сохранённый HH ID в том же пространстве справочника
- **THEN** система обновляет существующую запись и сохраняет её прежний UUID primary key

### Requirement: Реестр состояния синхронизации
Система MUST вести отдельное состояние для `dictionaries`, `areas`, `countries`, `professional_roles`, `industries`, `metro` и `languages`, включая UUID primary key `id`, обязательное уникальное логическое `name`, время последней попытки, время последней успешной полной синхронизации, текущий результат и безопасное описание ошибки. Поиск и upsert состояния MUST выполняться по уникальному `name`, не используя UUID как логическое имя снимка.

#### Scenario: Успешная синхронизация
- **WHEN** снимок справочника полностью применён и DB-транзакция зафиксирована
- **THEN** состояние этого справочника имеет UUID `id`, отмечается успешным и `last_success_at` получает время завершения

#### Scenario: Ошибка до commit
- **WHEN** загрузка, валидация или применение снимка завершается ошибкой
- **THEN** ошибка фиксируется для строки с тем же уникальным `name`, а её UUID `id` и прежний `last_success_at` не изменяются

#### Scenario: Повторное создание состояния
- **WHEN** scheduler, worker или seed повторно обеспечивает наличие состояния для существующего логического имени
- **THEN** новая строка не создаётся и существующий UUID primary key сохраняется

### Requirement: Идемпотентное применение полного снимка
Синхронизация каждого логического справочника MUST в одной DB-транзакции выполнять upsert по HH ID в пространстве этого справочника, обновлять изменившиеся поля, устанавливать `active=true` присутствующим элементам и `active=false` ранее известным элементам, отсутствующим в новом полном снимке. Повтор того же снимка MUST приводить к тому же состоянию без дубликатов и без замены UUID существующих записей.

#### Scenario: HH изменил название
- **WHEN** полный снимок содержит существующий HH ID с новым названием
- **THEN** запись обновляется без создания нового элемента, сохраняет UUID и остаётся активной

#### Scenario: Элемент исчез из снимка
- **WHEN** ранее активный HH ID отсутствует в валидном полном снимке своего справочника
- **THEN** запись и её UUID сохраняются, но она получает `active=false`

#### Scenario: Элемент вернулся
- **WHEN** ранее неактивный HH ID снова присутствует в полном снимке
- **THEN** его поля обновляются, прежний UUID сохраняется и `active` становится `true`

#### Scenario: Повторная доставка задачи
- **WHEN** worker повторно применяет уже успешно применённый снимок
- **THEN** уникальные записи, UUID, связи и активность не изменяются семантически и дубликаты не появляются
