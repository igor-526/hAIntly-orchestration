## Context

Текущая реализация `vacancy-service` хранит локальные PK таблиц `dictionary_items`, `areas`, `countries`, `professional_role_categories`, `professional_roles`, `industries`, `metro_cities`, `metro_lines`, `metro_stations` и `languages` как integer. В `dictionary_sync_states` поле `name` одновременно является логическим именем и PK. Публичное чтение уже адресует элементы по HH ID, а синхронизация выполняет типизированную загрузку семи полных снимков через HH API и upsert по предметным ключам.

Изменение выполняется до появления зависимых пользовательских данных, поэтому для локальной/test БД допустимо осознанное пересоздание таблиц. Production нельзя автоматически очищать. Локальная `docs/hh_openapi.yml` и официальный Redoc на момент проектирования подтверждают общедоступные GET `/dictionaries`, `/areas`, `/areas/countries`, `/professional_roles`, `/industries`, `/metro` и `/languages`, обязательный `HH-User-Agent` и изменяемость значений справочников.

Владелец схемы, данных и seed-процесса — `vacancy-service`. Инициатор seed — оператор/CI после миграций, получатель внешних HTTPS-запросов — HH API, запись выполняется только в PostgreSQL `vacancy-service`. Межсервисные HTTP, SSE и NATS-потоки не добавляются; существующее чтение `main-be` → `vacancy-service` остаётся без изменения `X-User-Id` и service-auth границы.

## Goals / Non-Goals

**Goals:**

- использовать UUID primary key во всех реляционных таблицах справочников и в `dictionary_sync_states`;
- сохранить HH ID и имя снимка как отдельные устойчивые domain keys и не менять адресацию LIST/GET API;
- предоставить отдельную идемпотентную seed-команду, использующую штатный типизированный HH sync и ожидающую terminal success всех семи снимков;
- определить воспроизводимый порядок local/test reset → upgrade → seed → runtime-проверки;
- исключить автоматический destructive reset production и смешивание внешних данных с Alembic.

**Non-Goals:**

- перенос или сохранение существующих локальных строк справочников при reset;
- статические HH payload/fixtures как production seed, изменение семантики sync, Beat или API;
- сброс БД других сервисов, изменение пользовательских данных, UI или фильтров вакансий;
- автоматический seed при каждом старте HTTP-приложения.

## Decisions

### 1. UUID является внутренним surrogate key, HH ID остаётся domain key

Каждая перечисленная таблица получает `id UUID PRIMARY KEY` с генерацией UUID приложением и/или PostgreSQL default, согласованной с принятым в репозитории подходом. `dictionary_sync_states.id` также становится UUID PK, а `name` — `NOT NULL UNIQUE`. Уникальность существующих HH domain keys сохраняется: `(dictionary_code, hh_id)` для общего справочника, `(category_hh_id, hh_id)` для профессиональных ролей и `hh_id` в остальных согласованных пространствах.

Текущие иерархические связи по HH domain keys можно сохранить, если они уже обеспечивают корректный sync и API; если реализация переводит их на UUID FK, она обязана сохранить отдельные HH-поля и порядок upsert родителей до детей. Во всех случаях внешняя адресация GET остаётся по HH ID, внутренний UUID не публикуется как обязательный route key.

Альтернатива — использовать HH ID как PK — отвергнута: тип и пространство внешнего идентификатора принадлежат HH, а составные ключи разных наборов не дают единого внутреннего подхода. Случайные integer PK также отвергнуты как несогласованные с требованием изменения.

### 2. Пересоздание выполняется миграцией, destructive запуск — только явно

Alembic head должен описывать итоговую UUID-схему с корректными PK, unique/FK и индексами. Поскольку текущие данные признаны одноразовыми, реализация может заменить ещё не выпущенные миграции или добавить миграцию пересоздания таблиц — выбор делается после проверки фактической истории deployment. В обоих случаях `alembic upgrade head` на чистой БД воспроизводимо создаёт схему.

Destructive reset оформляется отдельной явно вызываемой local/test командой с защитой по окружению/режиму и целевой БД `vacancy-service`. Она не входит в app startup, migration container production или обычный `upgrade`; при production-like окружении команда обязана отказать до DROP. Предпочтителен reset всей локальной/test схемы БД сервиса с последующим upgrade, если это не затрагивает чужие данные; узкий drop только набора таблиц допустим при сохранении целостности Alembic version state.

Альтернатива — безусловный DROP в Alembic upgrade — отвергнута из-за риска потери production-данных. Data migration integer→UUID не требуется для указанного reset-сценария и добавила бы ложное обещание сохранения строк.

### 3. Seed — оркестратор существующей синхронизации, а не источник payload

Отдельный CLI/module entrypoint после успешного `upgrade head` инициирует синхронизацию ровно семи логических снимков через существующие типизированный HH-адаптер и сервис применения. Seed не вызывает HTTP endpoint самого сервиса и не зависит от Beat. Он либо выполняет синхронизации непосредственно с теми же lock/transaction invariants, либо ставит штатные Celery-задачи и ограниченно ожидает их terminal state; конкретный способ выбирается по тестируемости без дублирования бизнес-логики.

Команда завершается с code 0 только когда все семь `dictionary_sync_states` имеют terminal success, непросроченный `last_success_at` от текущего запуска/подтверждённого идемпотентного результата и данные применены. Ошибка, timeout или неполный набор дают ненулевой code и перечень имён/безопасных классов ошибок без payload и секретов. Повторный запуск использует upsert и не создаёт дубликаты или новые UUID для уже существующих domain keys.

Статические JSON/YAML-ответы HH не коммитятся. Alembic отвечает только за схему и не выполняет сетевые запросы. Автосидирование в lifespan отвергнуто: оно делает health приложения зависимым от HH и скрывает операционный этап.

### 4. Операционный порядок фиксирован и проверяем

Для local/test rebuild применяется последовательность: остановить процессы записи `vacancy-service`; явно подтвердить безопасное окружение; reset только его БД/таблиц; `alembic upgrade head`; запустить seed; проверить schema constraints, семь sync states, непустые ожидаемые таблицы и отсутствие дублей; поднять app/worker/Beat; выполнить direct smoke `vacancy-service` и авторизованный proxy smoke через `main-be`.

Production deployment не выполняет reset: только `upgrade head`, затем отдельно контролируемый seed. Конфигурация HH URL, `HH-User-Agent`, timeout/retry, PostgreSQL, Redis/Celery берётся из уже валидируемого окружения без hardcode. Seed не логирует credentials, Redis URL, заголовки или полные ответы.

### 5. Обратная совместимость проверяется на API-границе

Схемы LIST/GET и route params не должны раскрывать необходимость знать UUID: запрос существующего активного элемента по HH ID возвращает тот же ресурс, неизвестный/неактивный — прежний `404`. `main-be` продолжает передавать `X-User-Id` и согласованный service credential по существующему контракту. Персональные данные не создаются и UUID пользователя seed-процессу не нужен.

## Risks / Trade-offs

- [Reset направлен не в ту БД] → explicit local/test guard, вывод безопасного идентификатора окружения, отказ в production-like режиме и отсутствие reset в автоматическом deployment.
- [Alembic version state не соответствует dropped таблицам] → reset и upgrade рассматриваются как одна документированная операция; тест начинается с чистой БД и проверяет `alembic current=head`.
- [Seed частично успешен из-за HH/сети] → независимые транзакции сохраняются, но общий exit code ненулевой; повторный запуск безопасно продолжает idempotent sync.
- [Повтор seed меняет внутренний UUID] → upsert выбирает существующую строку по domain key и не обновляет PK.
- [Конкуренция seed с Beat/worker] → использование существующего Redis lock на логический снимок и ограниченное ожидание/диагностируемое завершение.
- [HH изменяет значения во время теста] → проверки опираются на контракт, непустоту, уникальность и terminal success, а не на закоммиченный полный payload.

## Migration Plan

1. Обновить модели и Alembic head, добавить тесты UUID PK, unique/FK и чистого `upgrade head`.
2. Реализовать отдельный seed entrypoint и команды Makefile/compose, не включая reset или seed в startup приложения.
3. На изолированной local/test БД остановить процессы записи, выполнить guarded reset, `upgrade head`, seed и DB-проверки.
4. Пересобрать и поднять изменённые контейнеры; проверить health, worker/Beat и логи без утечек.
5. Выполнить повторный seed, подтвердить сохранение UUID по domain keys и отсутствие дублей; затем инициировать штатную повторную синхронизацию и проверить terminal success.
6. Выполнить direct API smoke и авторизованный proxy smoke через `main-be` по HH ID.

Rollback local/test: повторно развернуть предыдущий код и пересоздать одноразовую БД по предыдущему Alembic head, затем выполнить прежнюю синхронизацию. Production rollback не должен полагаться на destructive downgrade; до production rollout требуется backup и отдельное одобрение стратегии сохранения данных, если там уже существуют строки.

## Open Questions

Нет открытых продуктовых вопросов. Способ генерации UUID default и выбор direct orchestration либо Celery orchestration для seed определяются по существующим инфраструктурным паттернам, сохраняя нормативные требования к идемпотентности, locks и terminal success.
