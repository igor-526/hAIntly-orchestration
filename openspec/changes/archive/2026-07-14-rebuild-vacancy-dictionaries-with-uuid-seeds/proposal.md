## Why

Текущие таблицы справочников `vacancy-service` используют целочисленные внутренние ключи, а `dictionary_sync_states` использует логическое имя как primary key. До появления зависимых продуктовых данных структуру нужно привести к единому UUID-подходу проекта и сделать первичное наполнение после чистой миграции явным, воспроизводимым и проверяемым процессом.

## What Changes

- **BREAKING** для локальной и тестовой БД `vacancy-service`: существующие таблицы справочников и состояния синхронизации пересоздаются без сохранения текущих локальных строк; автоматический destructive reset production запрещён.
- Внутренний primary key каждой реляционной таблицы справочников становится UUID. Идентификаторы HH сохраняются отдельными domain keys с прежними уникальными ограничениями и внешними связями.
- `dictionary_sync_states` получает UUID primary key `id`, а `name` остаётся обязательным уникальным логическим именем снимка.
- Добавляется отдельная идемпотентная seed-команда `vacancy-service`: после применения Alembic-миграций она запускает существующую типизированную синхронизацию семи справочников через актуальный HH API, ожидает terminal success каждого снимка и завершается ошибкой при неполном результате.
- Seed не содержит закоммиченных payload HH, не подменяет сетевой адаптер и не смешивает данные со schema migration; повторный запуск безопасно обновляет тот же набор по domain keys.
- Фиксируется управляемый локальный/test порядок восстановления: reset таблиц/БД `vacancy-service` → `alembic upgrade head` → seed → проверка синхронизации, API и runtime. Production использует только недеструктивный upgrade и отдельно запускаемый seed.
- Публичные и внутренние LIST/GET-контракты остаются совместимыми: ресурс по-прежнему адресуется HH ID, локальный UUID не становится обязательным параметром API.
- В scope не входят статические снимки HH в репозитории, изменение расписания Beat, UI, фильтры вакансий, данные других сервисов и сброс их БД.

## Capabilities

### New Capabilities

- `vacancy-dictionary-seeding`: воспроизводимый идемпотентный запуск штатной HH-синхронизации после миграций с ожиданием и проверкой terminal success.

### Modified Capabilities

- `vacancy-dictionary-sync`: реляционные записи и состояния синхронизации получают внутренние UUID PK при сохранении HH domain keys и атомарного sync-поведения.
- `local-compose-runtime`: локальный/test runtime получает явный безопасно ограниченный reset/upgrade/seed workflow без автоматического destructive действия в production.

## Impact

- `vacancy-service`: SQLAlchemy-модели, Alembic-история/миграции, внешние ключи, репозиторий синхронизации, seed entrypoint, Makefile/compose-команды и тесты.
- БД `vacancy-service`: локальные и тестовые таблицы справочников пересоздаются; остальные сервисные БД не затрагиваются. HH ID и логические имена остаются стабильными предметными идентификаторами.
- HeadHunter API: seed использует тот же типизированный клиент и общедоступные GET-контракты `/dictionaries`, `/areas`, `/areas/countries`, `/professional_roles`, `/industries`, `/metro` и `/languages`; OAuth и статические payload не добавляются.
- `main-be`: реализация и публичный контракт не меняются, но proxy smoke входит в проверку обратной совместимости.
- `.docker-compose`, env-примеры и Makefile: при необходимости добавляются команды для явных upgrade/seed и локального/test reset; production compose не получает автоматический drop.
