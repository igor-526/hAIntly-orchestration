## Context

Сейчас `main-be` и `main-fe` имеют cookie-авторизацию пользователя, а `profile-service` уже существует как FastAPI/PostgreSQL-каркас с миграциями, healthcheck, compose/runtime и тестовой основой, но ещё не имеет контракта HH OAuth. Изменение пересекает три сервиса, добавляет внешнюю интеграцию с HH, отдельное владение персональными данными и токенами, а также открытый callback, поэтому требует явных решений по trust boundary.

Потоки изменения:


| Поток                                                 | Владелец данных                                                               | Инициатор                            | Получатель                                     | Протокол                              |
| ----------------------------------------------------- | ----------------------------------------------------------------------------- | ------------------------------------ | ---------------------------------------------- | ------------------------------------- |
| Начало HH OAuth                                       | `main-be` владеет пользователем и `state`; `profile-service` — HH-интеграцией | аутентифицированный `main-fe`        | `main-be` → `profile-service` → HH             | HTTP                                  |
| Завершение HH OAuth                                   | `profile-service` владеет связью, профилем и токенами                         | открытая callback-страница `main-fe` | `main-be` → `profile-service` → HH             | HTTP                                  |
| Чтение/удаление HH-аккаунтов                          | `profile-service`                                                             | аутентифицированный `main-fe`        | `main-be` → `profile-service`                  | HTTP                                  |
| Выбор активного аккаунта                              | `main-be` владеет выбранным идентификатором связи                             | аутентифицированный `main-fe`        | `main-be`, с проверкой через `profile-service` | HTTP                                  |


SSE, NATS JetStream и фоновая обработка в этих потоках не участвуют.

## Goals / Non-Goals

**Goals:**

- Реализовать мультиаккаунт HH с безопасным OAuth и хранением полного снимка профиля.
- Оставить браузер на границе `main-be`, а HH-токены — исключительно в `profile-service`.
- Использовать для межсервисных HTTP-вызовов локальную доверенную сеть без отдельной аутентификации и передавать пользовательский UUID как данные typed-контракта.
- Дать пользователю popup-поток подключения и устойчивые состояния выбора/отсутствия активного аккаунта.
- Сохранить действующие секреты `.env` при добавлении конфигурации и шаблонов.

**Non-Goals:**

- Резюме, вакансии, AI, уведомления и выдача HH-токенов другим сервисам.
- Межсервисная аутентификация, service key, mTLS и модель прав отдельных сервисов.
- Реализация пунктов «Настройки» и «Выход», содержимого карточек вакансий и AI-действий.
- Точная поддержка всех допустимых вариаций зарегистрированного redirect URI: приложение использует один явно настроенный абсолютный URI.



## Decisions



### 1. Владение данными разделяется по сервисным границам

`profile-service` хранит собственный UUID связи, UUID пользователя HAIntly, уникальный HH user id, полный JSON-снимок данных endpoint профиля, зашифрованные access/refresh token, срок access token и технические timestamps. Одна запись HH не может принадлежать двум пользователям HAIntly; один пользователь HAIntly имеет несколько записей.

`main-be` хранит только nullable UUID выбранной записи. Межбазового foreign key нет: при выборе `main-be` синхронно проверяет принадлежность через `profile-service`, а при удалении выбранного аккаунта очищает значение либо выбирает детерминированный оставшийся аккаунт. Альтернатива — хранить связи и токены в `main-be` — нарушает утверждённое владение `profile-service`.

### 2. Токены шифруются на repository boundary

Адаптер шифрования на базе `cryptography` получает `HH_TOKEN_ENCRYPT_KEY`, шифрует каждый токен перед записью и расшифровывает только при выполнении HH-запроса. DTO, логи, исключения и API никогда не содержат plaintext token. Сервис не стартует с отсутствующим/невалидным ключом; автоматическая генерация ключа запрещена, чтобы не сделать сохранённые данные нечитаемыми.

Альтернатива с шифрованием диска/БД недостаточна: приложение и дамп колонки всё равно раскрывали бы токены.

### 3. `state` выпускает и проверяет `main-be`

Аутентифицированная операция старта в `main-be` создаёт криптографически подписанный контекст с UUID пользователя, случайным nonce, временем истечения и назначением HH OAuth. `profile-service` использует непрозрачный для него `state` при построении URL через HH-адаптер. Открытый callback в `main-be` проверяет подпись, назначение, срок и одноразовость nonce, после чего передаёт проверенный UUID пользователя как поле typed-запроса вместе с code и redirect URI в `profile-service` для обмена токенов и загрузки профиля.

Nonce потребляется атомарно до обмена либо резервируется с безопасным финальным состоянием, исключающим повторное создание связи одним callback. Ошибки callback возвращают безопасный результат без содержимого `state`, code и токенов. Альтернатива с простым UUID пользователя в `state` отклонена как подмена привязки; зависимость callback от cookie отклонена, потому что popup может завершаться без доступной сессии.

### 4. HH изолируется асинхронным HTTP-адаптером

`profile-service` обращается к authorization/token/profile операциям HH только через интерфейс адаптера на `aiohttp`. Redirect URI, client credentials и базовые URL задаются окружением; production-код не выводит их секретные части в логи. Обмен code выполняется один раз, затем профиль загружается и сохраняется транзакционно; повторная привязка того же HH user id тому же владельцу обновляет профиль и токены, а другому владельцу отклоняется.

Прямые вызовы HH из `main-be` или `main-fe` отклонены: это размывает ответственность и раскрывает интеграционные детали.

### 5. Публичный API остаётся только у `main-be`

`main-fe` вызывает cookie-защищённые операции `main-be` для старта OAuth, списка, чтения, удаления и выбора аккаунта. Только операция завершения OAuth доступна без cookie и принимает code/state после redirect. `main-be` получает пользовательский UUID из cookie-сессии либо проверенного `state` и передаёт его в `profile-service` как типизированное поле внутреннего запроса; присланный браузером пользовательский идентификатор не используется.

Внутренний контракт `profile-service` включает построение authorization URL, завершение обмена для проверенного пользователя, LIST/GET/DELETE аккаунтов. Конкретные пути фиксируются при реализации единым typed client/router contract и contract tests, без прямого доступа к БД.

### 6. Межсервисные вызовы доверяют локальной сети

Вызовы `main-be` → `profile-service` и будущие вызовы других микросервисов → `main-be` выполняются без service key, cookie или `X-User-Id`: доступ к этим контрактам ограничивается локальной сетью deployment. Когда операции нужен пользовательский контекст, UUID является обычным валидируемым полем typed-запроса, а не доказательством идентичности отправителя. Публичные browser routes `main-be` продолжают использовать cookie либо проверенный OAuth `state` и не принимают произвольный user id клиента.

Это осознанная граница текущего deployment. Альтернатива с service key отклонена по прямому уточнению требований; при выходе сервисов за пределы доверенной сети потребуется отдельный OpenSpec change для service authentication.

### 7. Popup сообщает результат контролируемо

Главная страница открывает небольшое окно на URL, полученный от `main-be`. После HH redirect открытый маршрут `main-fe` передаёт query в callback `main-be`, показывает success/error и при успехе уведомляет только ожидаемый opener через `postMessage` с проверяемым origin, затем закрывается. Opener обновляет список и активный аккаунт; блокировка popup и ручное закрытие получают понятное состояние.

Callback-страница не показывает защищённый контент и не становится универсальным исключением route guard. Токены/code/state не сохраняются в browser storage и не логируются.

### 8. Главная страница вводится как shell будущих функций

Шапка содержит текстовый логотип, selector HH-аккаунта и профильное меню с пока неактивными пунктами. HH selector реализуется как доступная dropdown-кнопка: trigger показывает активный профиль, а открывающееся под ним меню содержит все профили с явным активным состоянием, выбор другого профиля, добавление и удаление активного аккаунта. Перед удалением UI открывает modal dialog с именем профиля, явными действиями отмены/подтверждения и управлением фокусом. При отсутствии аккаунтов вместо selector action и рабочих панелей показывается призыв подключить HH. При наличии активного аккаунта отображаются две панели с минимальной шириной и доступным drag-разделителем; вакансии и AI-кнопки являются неинтерактивным shell текущего scope.

Альтернатива с реализацией vacancy API в этом change отклонена как преждевременное смешение сервисных этапов.

### 9. Контракты, БД, безопасность и runtime

- **API:** добавляются публичные proxy-операции `main-be` и внутренний typed HTTP-контракт `profile-service` без отдельной межсервисной аутентификации.
- **События:** SSE subjects и NATS JetStream schemas не добавляются.
- **БД:** отдельная схема/миграция `profile-service`; миграция `main-be` для nullable active HH account id. Прямых межбазовых связей нет.
- **Фоновая обработка:** отсутствует; OAuth и CRUD синхронны.
- **Безопасность/ПД:** HH-профиль является персональными данными; ответы минимизируются до нужных UI-полей, удаление физически удаляет профиль и ciphertext токенов, секреты маскируются, callback/state имеют TTL и replay-защиту.
- **Инфраструктура:** существующие каркас `profile-service`, compose, БД, migration container, healthcheck и Makefile-команды переиспользуются; новые HH-переменные добавляются в локальную конфигурацию и tracked example без значений секретов, существующий `.env` не заменяется.
- **AI/LLM и broker:** не затрагиваются.

### 10. Адрес каждой HTTP-зависимости принадлежит окружению вызывающего сервиса

Каждый сервис получает base URL исходящего межсервисного HTTP-вызова только из собственного окружения. Для текущего направления `main-be` → `profile-service` обязательная переменная `PROFILE_SERVICE_URL` принадлежит `.env` `main-be`: production-код и compose не содержат default или зашитого адреса, а отсутствующее либо невалидное абсолютное HTTP(S)-значение останавливает сервис при загрузке настроек. Tracked `.env.example` может документировать безопасное локальное значение, но не является runtime fallback.

Это правило направленное: конфигурацию хранит потребитель зависимости, а не вызываемый сервис и не общий compose-файл. Если в будущем `profile-service` получит исходящий контракт к `main-be`, URL и предусмотренный тем отдельным контрактом service key будут принадлежать окружению `profile-service`; в текущем change такого обратного вызова нет, поэтому `MAIN_BE_URL` и service key не добавляются. Для уже реализованного вызова `main-be` → `profile-service` сохраняется решение об отсутствии отдельной межсервисной аутентификации.



## Risks / Trade-offs

- [Доверие локальной сети не защищает контракт при ошибочной публикации порта или компрометации соседнего сервиса] → не публиковать внутренние routes за пределы deployment network, проверить compose/network boundary; при изменении модели размещения спроектировать межсервисную аутентификацию отдельным change.
- [Сбой после погашения nonce, но до сохранения аккаунта потребует повторного входа] → возвращать явную повторяемую ошибку и не ослаблять replay-защиту ради автоматического retry.
- [Удаление аккаунта и обновление active id происходят в разных БД] → сделать proxy-операцию идемпотентной, очищать/перевыбирать active id после подтверждённого удаления и восстанавливать согласованность при следующем чтении списка.
- [Полный JSON профиля HH меняется без миграции] → хранить снимок как JSON с минимальным индексируемым набором стабильных колонок и валидировать обязательный HH id.
- [Popup может быть заблокирован браузером] → открывать его непосредственно из пользовательского действия и показывать инструкцию/повтор.
- [Неверный redirect URI ломает обмен code] → использовать одну конфигурационную переменную одинаково при построении URL и token exchange, валидировать абсолютный HTTP(S) URI на старте.



## Migration Plan

1. Расширить существующий каркас `profile-service`: добавить HH-зависимости и конфигурацию, миграцию аккаунтов, crypto/HH adapters и автономные тесты, сохранив готовый runtime.
2. Добавить OAuth state store и миграцию active account в `main-be`.
3. Добавить внутренний client/proxy contract и интеграционные contract tests между `main-be` и `profile-service`.
4. Добавить popup/callback и shell главной страницы `main-fe`, затем выполнить frontend tests/build.
5. Запустить миграции, сервисы и smoke OAuth в окружении с зарегистрированным redirect URI; секреты вводятся операционно, не коммитятся.
6. При rollback сначала отключить UI/proxy routes, затем откатить nullable колонку `main-be` и новую миграцию `profile-service`. БД `profile-service` сохранить до подтверждения, что связанные профили больше не нужны; удаление данных выполнять отдельно.



## Open Questions

Отсутствуют. Имена конкретных внутренних endpoint и формат минимального account DTO могут быть зафиксированы реализацией и contract tests без изменения описанных границ и поведения.
