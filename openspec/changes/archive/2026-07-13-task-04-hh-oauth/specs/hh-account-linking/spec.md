## ADDED Requirements

### Requirement: Profile-service владеет связями и профилями HH
`profile-service` MUST хранить для каждой связи собственный UUID, UUID пользователя HAIntly, уникальный идентификатор пользователя HH, полный предоставленный HH снимок профиля и технические timestamps в собственной PostgreSQL БД; один пользователь HAIntly MUST иметь возможность владеть несколькими HH-аккаунтами.

#### Scenario: Первый HH-аккаунт пользователя
- **WHEN** OAuth завершается профилем HH, ещё не связанным с другим пользователем HAIntly
- **THEN** `profile-service` создаёт связь и сохраняет полный снимок профиля в своей БД

#### Scenario: Несколько аккаунтов
- **WHEN** один пользователь HAIntly успешно подключает разные HH user id
- **THEN** каждая связь получает отдельный UUID и присутствует в списке этого пользователя

#### Scenario: HH-аккаунт уже принадлежит другому пользователю
- **WHEN** OAuth возвращает HH user id, связанный с другим пользователем HAIntly
- **THEN** `profile-service` отклоняет привязку без раскрытия данных владельца

### Requirement: HH-токены шифруются в покое
`profile-service` MUST шифровать access и refresh token через crypto-адаптер до записи, MUST расшифровывать их только на время разрешённого HH-запроса и MUST отказывать в старте при отсутствующем или невалидном `HH_TOKEN_ENCRYPT_KEY`.

#### Scenario: Сохранение полученных токенов
- **WHEN** HH возвращает access и refresh token
- **THEN** база получает только ciphertext и метаданные срока действия, а plaintext не попадает в DTO или логи

#### Scenario: Использование сохранённого токена
- **WHEN** разрешённая операция должна обратиться к HH
- **THEN** repository boundary расшифровывает токен в памяти и не сохраняет plaintext обратно

#### Scenario: Ключ шифрования не настроен
- **WHEN** `profile-service` запускается без валидного `HH_TOKEN_ENCRYPT_KEY`
- **THEN** сервис завершается конфигурационной ошибкой без генерации нового ключа

### Requirement: OAuth начинается в контексте аутентифицированного пользователя
`main-be` MUST разрешать старт HH OAuth только пользователю с действующей cookie-сессией, MUST выпустить ограниченный по времени подписанный `state` с UUID пользователя и случайным nonce и MUST получить authorization URL через адаптер `profile-service` с настроенным redirect URI.

#### Scenario: Успешный старт
- **WHEN** аутентифицированный пользователь начинает подключение HH
- **THEN** `main-be` возвращает URL HH с client context, redirect URI и непрозрачным подписанным `state`

#### Scenario: Гость начинает подключение
- **WHEN** запрос старта не имеет действующей пользовательской сессии
- **THEN** `main-be` отклоняет запрос и не выпускает `state`

### Requirement: Открытый callback безопасно завершает OAuth
Callback `main-be` MUST быть доступен без cookie, MUST проверить подпись, назначение, срок и одноразовый nonce `state`, MUST извлечь пользователя только из проверенного `state` и затем MUST поручить `profile-service` обмен code, получение профиля и сохранение связи.

#### Scenario: Успешный callback без cookie
- **WHEN** callback получает действительный code и неиспользованный `state` без cookie-сессии
- **THEN** `main-be` передаёт проверенный UUID пользователя в `profile-service`, который обменивает code, получает профиль и сохраняет либо обновляет его связь

#### Scenario: Подменённый или просроченный state
- **WHEN** подпись `state` неверна, назначение не совпадает либо срок истёк
- **THEN** callback отклоняется до обмена code и не создаёт связь

#### Scenario: Повтор callback
- **WHEN** тот же nonce callback используется повторно
- **THEN** `main-be` отклоняет повтор независимо от наличия действительного code

#### Scenario: HH возвращает ошибку
- **WHEN** callback содержит отказ пользователя либо HH token/profile операция завершается ошибкой
- **THEN** система возвращает безопасный ошибочный результат без code, state, токенов и внутренних деталей

### Requirement: Аккаунты доступны только владельцу через main-be
`main-be` MUST предоставлять аутентифицированному пользователю операции LIST, GET и DELETE как proxy к `profile-service`; `main-be` MUST передавать UUID текущего пользователя как типизированное поле внутреннего запроса, а `profile-service` MUST ограничивать каждую операцию этим UUID и MUST NOT возвращать HH-токены.

#### Scenario: Список аккаунтов
- **WHEN** пользователь запрашивает свои HH-аккаунты
- **THEN** система возвращает только принадлежащие ему связи с безопасными UI-полями профиля и без токенов

#### Scenario: Чтение чужого аккаунта
- **WHEN** пользователь запрашивает UUID связи, принадлежащей другому пользователю
- **THEN** система не возвращает профиль или признак существования чужой связи

#### Scenario: Удаление аккаунта
- **WHEN** владелец удаляет свою связь
- **THEN** `profile-service` физически удаляет профиль и ciphertext токенов, а повторное удаление не раскрывает дополнительные данные

### Requirement: Main-be хранит выбранный HH-аккаунт пользователя
`main-be` MUST хранить nullable UUID выбранной связи HH в собственной пользовательской модели, MUST разрешать выбрать только подтверждённую `profile-service` связь текущего пользователя и MUST поддерживать согласованное состояние после удаления.

#### Scenario: Выбор принадлежащего аккаунта
- **WHEN** пользователь выбирает UUID связи из своего списка
- **THEN** `main-be` проверяет принадлежность через `profile-service` и сохраняет UUID как активный

#### Scenario: Выбор чужого или отсутствующего аккаунта
- **WHEN** пользователь передаёт UUID чужой или отсутствующей связи
- **THEN** `main-be` отклоняет изменение и сохраняет прежний выбор

#### Scenario: Удаление выбранного аккаунта
- **WHEN** пользователь удаляет активную связь
- **THEN** `main-be` очищает выбор либо детерминированно выбирает одну из оставшихся принадлежащих связей

#### Scenario: У пользователя нет аккаунтов
- **WHEN** список связей пуст
- **THEN** выбранный UUID имеет значение null и защищённые HH-функции не считают контекст аккаунта доступным

### Requirement: HH-интеграция и redirect URI конфигурируются
`profile-service` MUST изолировать authorization, token и profile запросы HH интерфейсом асинхронного HTTP-адаптера и MUST использовать один валидированный абсолютный redirect URI из окружения при построении authorization URL и обмене code.

#### Scenario: Одинаковый redirect URI
- **WHEN** выполняются начало OAuth и последующий token exchange
- **THEN** адаптер передаёт одинаковое настроенное значение redirect URI в обе операции

#### Scenario: Невалидная конфигурация redirect URI
- **WHEN** redirect URI отсутствует либо не является допустимым абсолютным HTTP(S) URI
- **THEN** сервис отклоняет конфигурацию до начала OAuth

#### Scenario: Автономный тест HH-адаптера
- **WHEN** выполняются unit и contract tests
- **THEN** HH transport подменяется и тесты не требуют реальных credentials или сетевого доступа к HH

### Requirement: Новая конфигурация не уничтожает существующее окружение
Runtime-настройка существующего `profile-service` MUST добавлять переменные HH OAuth, redirect URI и `HH_TOKEN_ENCRYPT_KEY` без замены существующего `.env`; tracked example MUST содержать только безопасные placeholders, а существующие compose, БД, migration container, healthcheck и Makefile-команды MUST сохраняться.

#### Scenario: Обновление локального окружения
- **WHEN** добавляется конфигурация `profile-service`
- **THEN** существующие параметры приложения в `.env` сохраняются и дополняются требуемыми именами

#### Scenario: Проверка tracked env example
- **WHEN** репозиторий проверяется перед merge
- **THEN** example документирует все обязательные переменные и не содержит действительных HH credentials или ключей шифрования

#### Scenario: Существующий runtime сохраняется
- **WHEN** функциональность HH OAuth добавляется в `profile-service`
- **THEN** реализация использует существующие каркас сервиса, compose-файл, PostgreSQL, migration container, healthcheck и Makefile-команды без их повторного создания

### Requirement: Адрес исходящей сервисной зависимости задаётся окружением вызывающего сервиса
Каждый backend-сервис MUST получать base URL своей исходящей HTTP-зависимости из собственного окружения, MUST валидировать его как абсолютный HTTP(S) URL при старте и MUST завершаться конфигурационной ошибкой при отсутствующем или невалидном значении; production-код и compose MUST NOT задавать hardcoded/default адрес зависимости. Для текущего вызова `main-be` → `profile-service` переменная `PROFILE_SERVICE_URL` MUST принадлежать окружению `main-be` и MUST использоваться typed-клиентом без изменения кода или image. Обратный вызов `profile-service` → `main-be`, `MAIN_BE_URL` и service key не входят в текущий контракт и MUST NOT добавляться без отдельного требования.

#### Scenario: Main-be использует адрес profile-service из своего env
- **WHEN** `main-be` запускается с валидным `PROFILE_SERVICE_URL` в собственном окружении
- **THEN** его typed HTTP client отправляет запросы по этому base URL, а значение можно заменить на контролируемый endpoint без изменения кода или image

#### Scenario: Адрес зависимости отсутствует
- **WHEN** `main-be` запускается без `PROFILE_SERVICE_URL`
- **THEN** загрузка настроек завершается конфигурационной ошибкой до обслуживания запросов

#### Scenario: Адрес зависимости невалиден
- **WHEN** `PROFILE_SERVICE_URL` не является абсолютным HTTP(S) URL
- **THEN** загрузка настроек завершается конфигурационной ошибкой до обслуживания запросов

#### Scenario: Compose не подменяет env вызывающего сервиса
- **WHEN** production compose создаёт контейнер `main-be`
- **THEN** `PROFILE_SERVICE_URL` поступает из env-файла `main-be`, а service definition не содержит hardcoded/default адреса
