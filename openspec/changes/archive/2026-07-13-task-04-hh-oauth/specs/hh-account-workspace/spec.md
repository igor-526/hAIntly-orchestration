## ADDED Requirements

### Requirement: Подключение HH выполняется во всплывающем окне
`main-fe` MUST открывать полученный от `main-be` HH authorization URL во всплывающем окне непосредственно из пользовательского действия, MUST отслеживать успешное завершение и MUST обновлять список аккаунтов после него.

#### Scenario: Успешное подключение
- **WHEN** пользователь выбирает добавление аккаунта, проходит HH UI и callback завершается успешно
- **THEN** popup сообщает результат opener, закрывается, а главная страница повторно загружает аккаунты и активный выбор

#### Scenario: Popup заблокирован
- **WHEN** браузер блокирует создание окна
- **THEN** интерфейс показывает понятную ошибку и позволяет повторить действие

#### Scenario: Пользователь закрыл окно
- **WHEN** popup закрыт до успешного callback
- **THEN** интерфейс прекращает pending-состояние без ложного добавления аккаунта

### Requirement: Callback-страница завершает OAuth без пользовательской сессии
`main-fe` MUST предоставлять отдельный открытый callback-маршрут, который передаёт полученные HH query parameters только callback-операции `main-be`, показывает безопасный результат и MUST NOT сохранять или логировать code, state либо токены.

#### Scenario: Callback открыт без cookie
- **WHEN** HH перенаправляет popup на callback-страницу без действующей HAIntly cookie
- **THEN** страница не перенаправляет пользователя на login и запрашивает завершение OAuth у `main-be`

#### Scenario: Callback завершён успешно
- **WHEN** `main-be` подтверждает привязку
- **THEN** страница уведомляет только ожидаемый opener с проверкой origin и закрывает popup

#### Scenario: Callback завершён ошибкой
- **WHEN** `main-be` отклоняет callback или недоступен
- **THEN** страница показывает безопасное сообщение и возможность закрыть окно без отображения query-секретов

### Requirement: Шапка управляет HH-аккаунтами
На защищённой главной странице `main-fe` MUST показывать на всю ширину текстовый логотип «HAIntly», меню профиля и HH account selector в виде доступного dropdown с действиями добавления, выбора и удаления аккаунта; удаление MUST требовать подтверждения в modal dialog.

#### Scenario: Пользователь имеет несколько аккаунтов
- **WHEN** загружены две или более принадлежащие пользователю связи
- **THEN** trigger selector показывает активный аккаунт, а dropdown под trigger показывает все связи с явным active state и позволяет выбрать другую связь без возможности явно снять выбор

#### Scenario: Добавление аккаунта
- **WHEN** пользователь выбирает действие добавления
- **THEN** запускается popup-поток HH OAuth

#### Scenario: Удаление аккаунта
- **WHEN** пользователь подтверждает удаление принадлежащего аккаунта
- **THEN** UI вызывает proxy `main-be`, обновляет список и отражает новый активный либо пустой выбор

#### Scenario: Запрос подтверждения удаления
- **WHEN** пользователь выбирает удаление активного HH-аккаунта в dropdown
- **THEN** UI не отправляет DELETE-запрос, открывает доступный modal dialog с именем профиля и явными действиями «Отмена»/«Удалить»; Escape и отмена закрывают dialog, а pending исключает повторную отправку

#### Scenario: Меню профиля
- **WHEN** пользователь открывает меню профиля
- **THEN** интерфейс показывает пункты «Настройки» и «Выход» без реализации их бизнес-действий в текущем change

### Requirement: Пустое состояние требует подключить HH
Если у пользователя нет активного HH-аккаунта, `main-fe` MUST заменить кнопку открытия selector кнопкой входа в HH и MUST не показывать рабочие панели, вместо них выводя сообщение о необходимости подключить аккаунт.

#### Scenario: Новый пользователь без HH
- **WHEN** список аккаунтов пуст и active id равен null
- **THEN** пользователь видит действие входа в HH и пояснение, а vacancy workspace скрыт

#### Scenario: Удалён последний аккаунт
- **WHEN** пользователь удаляет последнюю связь
- **THEN** интерфейс переходит в то же пустое состояние без возможности сохранить несуществующий выбор

### Requirement: Рабочая область имеет две изменяемые панели
При наличии активного HH-аккаунта `main-fe` MUST показывать область до конца доступного экрана, разделённую на левую панель вакансий и правую панель выбранной вакансии, с минимальной шириной каждой панели и доступным drag-разделителем.

#### Scenario: Отображение shell
- **WHEN** активный аккаунт доступен
- **THEN** слева показаны controls фильтров, selector резюме и shell списка вакансий, а справа — shell информации и AI-действий

#### Scenario: Изменение ширины
- **WHEN** пользователь перемещает разделитель pointer или поддерживаемым клавиатурным действием
- **THEN** ширины панелей меняются в допустимых пределах и ни одна не становится уже установленного минимума

#### Scenario: Узкий viewport
- **WHEN** viewport не позволяет разместить обе панели с минимальной шириной
- **THEN** интерфейс использует адаптивное размещение без недоступных controls и неконтролируемого горизонтального переполнения страницы

### Requirement: Shell не имитирует отсутствующие продуктовые операции
До появления контрактов вакансий и AI `main-fe` MUST отображать элементы вакансий, резюме и AI только как явно неактивный shell и MUST NOT отправлять запросы напрямую в `vacancy-service`, `ai-service` или HH.

#### Scenario: Пользователь взаимодействует с будущим действием
- **WHEN** пользователь видит ещё не реализованную кнопку фильтров или AI-действие
- **THEN** control обозначен недоступным и не инициирует сетевой запрос
