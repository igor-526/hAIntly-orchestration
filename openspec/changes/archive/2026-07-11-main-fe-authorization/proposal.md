## Why

Пустому приложению `main-fe` нужен первый пользовательский сценарий: регистрация и вход через уже существующий cookie-based контракт `main-be`, а также недоступность главной страницы для неаутентифицированного пользователя. Это создаёт безопасную основу для последующих экранов HAIntly и устраняет необходимость хранить токены в браузерном JavaScript.

## What Changes

- Добавляется адаптивная страница `/login` с надписью «HAIntly», полноширинным переключателем «Вход»/«Регистрация», соответствующей формой и кнопкой отправки.
- Вход принимает email или логин и пароль, вызывает публичный HTTP API `main-be`, передавая cookie credentials, и после успеха переводит пользователя на `/`.
- Регистрация принимает email, логин и пароль, отражает клиентские и серверные ошибки, а после успешного создания пользователя переводит форму в режим входа без автоматического создания сессии.
- Добавляется проверка текущей cookie-сессии, однократное восстановление access cookie через refresh и защита `/` с перенаправлением гостя на `/login`; аутентифицированный пользователь, открывший `/login`, перенаправляется на `/`.
- Главная страница `/` заменяется минимальной заглушкой для аутентифицированного пользователя.
- Формируется слоистая frontend-структура `page/route -> feature UI -> feature hook -> feature service -> API client -> main-be`, типизированная граница API и тестовая инфраструктура без live backend.
- DEV-команда `main-fe` настраивается на порт `3100`.
- Подтверждение email, восстановление пароля, выход из системы, HH OAuth и содержимое будущей главной страницы явно не входят в scope изменения. Хотя общий продуктовый поток предусматривает подтверждение email, текущий публичный контракт `main-be` не предоставляет такого endpoint, поэтому change его не имитирует.
- Контракт, реализация и владение пользовательскими данными в `main-be` не изменяются; frontend использует существующие `POST /api/auth/register`, `POST /api/auth/token`, `POST /api/auth/refresh` и `GET /api/auth/verify`.

## Capabilities

### New Capabilities

- `frontend-user-auth`: пользовательский интерфейс регистрации и входа, cookie-based API boundary, состояние сессии и маршрутизация между `/login` и защищённой `/`.

### Modified Capabilities

Отсутствуют.

## Impact

- `main-fe`: новые маршруты, auth feature, типизированный HTTP-клиент, управление состоянием формы и сессии, адаптивная стилизация, конфигурация DEV-порта и frontend-тесты.
- `main-be`: используется без изменений как владелец пользователей и аутентификации; change фиксирует потребление его существующего публичного HTTP-контракта и HttpOnly cookie.
- Публичная браузерная интеграция требует корректного `NEXT_PUBLIC_API_BASE_URL` и разрешённого origin `http://localhost:3100` в текущей конфигурации CORS `main-be`; значения backend-конфигурации не меняются в рамках этого change.
- `profile-service`, `vacancy-service`, `ai-service`, `notification-service`, SSE, NATS JetStream, базы данных и фоновые процессы не затрагиваются.
