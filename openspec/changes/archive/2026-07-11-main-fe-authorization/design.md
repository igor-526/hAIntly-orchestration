## Context

`main-fe` — стандартная пустая заготовка Next.js 16 App Router с React 19, TypeScript и Tailwind CSS 4; auth-слоя и тестового стека пока нет. `main-be` уже владеет пользователями и сессиями: регистрация принимает JSON `{email, username, password}`, вход — `{login, password}`, а access/refresh JWT выдаются только как HttpOnly cookie. Проверка сессии возвращает пользователя, refresh обновляет обе cookie. Ошибки имеют поле `detail`, причём validation detail может быть массивом.

Референсные frontend-проекты подтверждают полезность выделенных API client, feature service и hooks, но их конкретные UI-библиотеки, глобальные store и исторические auth-решения не переносятся автоматически. У HAIntly пока две страницы и отсутствует необходимость в глобальном state manager.

Потоки изменения:


| Поток                 | Владелец данных | Инициатор                                        | Получатель                          | Протокол                  |
| --------------------- | --------------- | ------------------------------------------------ | ----------------------------------- | ------------------------- |
| Регистрация           | `main-be`       | форма регистрации `main-fe`                      | `POST /api/auth/register` `main-be` | HTTP с JSON               |
| Вход и выдача сессии  | `main-be`       | форма входа `main-fe`                            | `POST /api/auth/token` `main-be`    | HTTP с JSON и Set-Cookie  |
| Проверка сессии       | `main-be`       | auth provider/gate `main-fe`                     | `GET /api/auth/verify` `main-be`    | HTTP с cookie credentials |
| Восстановление сессии | `main-be`       | API boundary `main-fe` после неуспешной проверки | `POST /api/auth/refresh` `main-be`  | HTTP с refresh cookie     |


SSE и NATS JetStream в этих потоках не участвуют.

## Goals / Non-Goals

**Goals:**

- Реализовать понятный адаптивный UX регистрации и входа на `/login` и защищённую заглушку `/`.
- Сохранить цепочку `route -> feature UI -> hook -> service -> API client -> main-be` и типизировать входные/выходные данные.
- Использовать HttpOnly cookie `main-be` без копирования токенов в JavaScript-хранилища.
- Покрыть validation, loading, success и backend/network error тестами без live backend.
- Запускать `npm run dev` на порту `3100`.

**Non-Goals:**

- Изменение endpoint, схем, cookie-политики, БД или backend-кода `main-be`.
- Подтверждение email, восстановление пароля, logout, роли/permissions и управление профилем.
- HH OAuth, SSE, уведомления и прочие продуктовые сценарии после авторизации.
- Полноценное содержимое главной страницы и отдельная дизайн-система.



## Decisions



### 1. Feature-слои без глобального state manager

Auth feature разделяется на типы/validators, API boundary, чистый service, hooks и UI. Небольшой React provider хранит только вычисленное состояние текущего пользователя (`checking`, `authenticated`, `anonymous`) и предоставляет повторную проверку после входа. Page-файлы остаются композиционными.

Это сохраняет расширяемую архитектуру референсов, но не вводит Zustand/MobX для двух маршрутов. Альтернатива — выполнять fetch прямо в странице — отклонена из-за смешения UI, сетевого контракта и orchestration.

### 2. Управляемые React-формы и чистая валидация

При отсутствии принятого form-стека используются controlled inputs и отдельные чистые функции validation. Вход требует непустые `login` и `password`; регистрация зеркалирует известные ограничения `main-be`: корректный email, логин длиной 3–32 с допустимым шаблоном, пароль длиной 8–128 как минимум с одной буквой и одной цифрой. Клиентская проверка улучшает UX, но серверный ответ остаётся окончательным.

Альтернатива — добавить React Hook Form и Zod — не выбрана: для двух компактных форм стоимость новых runtime-зависимостей выше выгоды. Решение можно пересмотреть при появлении сложных форм.

### 3. Единый fetch boundary с cookie credentials

API client строит URL из обязательной публичной переменной `NEXT_PUBLIC_API_BASE_URL`, нормализует успешные пустые ответы, JSON и `detail` ошибок и всегда использует `credentials: "include"`. Feature service знает точные auth endpoints и DTO, но не React.

Токены не читаются, не логируются и не сохраняются в local/session storage. Альтернатива с Bearer token отклонена, поскольку противоречит фактическому HttpOnly-cookie контракту `main-be`.

### 4. Ограниченное восстановление сессии

При инициализации provider вызывает verify. Только при `401` он один раз вызывает refresh, затем один раз повторяет verify. Параллельные попытки refresh объединяются одним promise; циклический retry запрещён. Сетевые и серверные ошибки не трактуются как неверные credentials: gate показывает состояние ошибки и безопасный повтор, а не бесконечный redirect.

Альтернатива — refresh любого запроса общим бесконечным interceptor — отклонена из-за риска циклов и неявных повторов небезопасных команд. В этом change восстановление нужно только для проверки сессии.

### 5. Client-side route gate с явным checking-состоянием

Общий provider подключается в root layout. Защищённая композиция `/` до завершения verify показывает нейтральный loading state, затем рендерит заглушку либо выполняет `router.replace('/login')`. `/login` аналогично перенаправляет подтверждённого пользователя на `/`. Это исключает показ защищённого содержимого до проверки.

Next middleware не используется: он не должен самостоятельно декодировать backend JWT, а наличие access cookie не доказывает её валидность. Полностью server-side verify потребовал бы дополнительного BFF/proxy-решения, отсутствующего в scope.

### 6. Поведение регистрации и входа

Успешный вход (`204`) запускает повторную проверку пользователя и только после успешного verify переводит на `/`. Успешная регистрация (`201`) не считается аутентификацией, так как endpoint не устанавливает cookie: UI показывает подтверждение, переключается во «Вход» и сохраняет введённый логин для удобства. Автоматический вход отклонён как скрытый второй сетевой шаг, которого пользователь не задал и backend-контракт регистрации не обещает.

Submit блокируется на время запроса, field errors отображаются рядом с полями, общая backend/network ошибка — отдельно и доступно для assistive technologies. Переключение режима очищает пароль, field и submit errors.

### 7. Адаптивный UI и DEV-порт

Страница входа центрирует карточку по обеим осям. На широком экране карточка имеет разумный `max-width`, на мобильном занимает доступную ширину с внешними отступами. Переключатель занимает всю ширину формы; логотип временно представлен текстом «HAIntly». Используются семантические HTML-элементы и Tailwind, без UI-kit.

Скрипт `dev` фиксируется как `next dev --port 3100`. API base URL остаётся конфигурацией окружения; `main-fe` не угадывает порт `main-be`.

### 8. Тестовый контур

Добавляются Vitest, jsdom и React Testing Library. API boundary тестируется mock fetch, validators/services/hooks — детерминированно, UI и маршрутизация — с подменой Next router и auth API. Тесты не обращаются к live backend. Обязательные команды проверки: lint, unit/component tests и production build.

### 9. Контракты, данные и инфраструктура

- **API:** backend endpoint и DTO не меняются; frontend добавляет их типизированных потребителей.
- **События:** SSE и NATS events не добавляются и не меняются.
- **БД:** миграций и изменений владения данными нет; пользовательские данные принадлежат `main-be`.
- **Фоновая обработка:** отсутствует.
- **Безопасность и персональные данные:** пароль существует только в локальном состоянии формы на время ввода, очищается при смене режима и не логируется; токены доступны только браузеру как HttpOnly cookie; email/логин не пишутся в console/debug UI. Cookie-запросы требуют точного разрешённого origin и `allow_credentials` на `main-be`.
- **HH, AI/LLM, брокер:** не затрагиваются, адаптеры не требуются.



## Risks / Trade-offs

- [Origin `http://localhost:3100` отсутствует в CORS-конфигурации окружения `main-be`] → документировать требуемую переменную окружения и проверить интеграцию в Quality Gate; не расширять backend scope скрытым изменением.
- [Client-side gate кратко показывает loading вместо server-rendered страницы] → не рендерить содержимое `/` до успешного verify; BFF/server auth рассматривать отдельным change.
- [Backend validation detail бывает строкой или массивом] → нормализовать оба формата в безопасное пользовательское сообщение и сохранить field mapping только для известных полей.
- [Refresh cookie ограничена путём `/api/auth`] → вызывать фактический `/api/auth/refresh` через настроенный base URL и покрыть сериализацию тестом.
- [Автоматический refresh пока охватывает только bootstrap сессии] → расширять retry policy отдельным решением при появлении остальных защищённых API, особенно для изменяющих состояние команд.
- [Нет подтверждения email несмотря на общий продуктовый pipeline] → явно не изображать подтверждение в UI; сначала добавить контракт `main-be` отдельным OpenSpec change.



## Migration Plan

1. Добавить тестовый стек и auth-слои `main-fe`, сохранив конфигурируемый URL `main-be`.
2. Заменить стартовую страницу, добавить `/login`, provider/gates и DEV-порт `3100`.
3. Выполнить lint, tests и build; затем проверить браузерный сценарий с `main-be`, где разрешён origin `http://localhost:3100`.
4. Rollback выполняется возвратом frontend change: backend-контракт и данные не мигрируются, поэтому отдельного отката данных нет.
