## Purpose

TBD — требования к пользовательской авторизации в `main-fe`.

## Requirements

### Requirement: Страница входа и регистрации
`main-fe` MUST предоставлять маршрут `/login` с текстовым логотипом «HAIntly», полноширинным переключателем режимов «Вход» и «Регистрация», формой текущего режима и соответствующей кнопкой отправки.

#### Scenario: Первое открытие страницы входа
- **WHEN** неаутентифицированный пользователь открывает `/login`
- **THEN** система показывает режим «Вход» с полями логина и пароля и кнопкой «Войти»

#### Scenario: Переключение на регистрацию
- **WHEN** пользователь выбирает «Регистрация»
- **THEN** система показывает поля email, логина и пароля и кнопку «Зарегистрироваться», а переключатель остаётся на всю ширину формы

#### Scenario: Сброс чувствительного состояния при переключении
- **WHEN** пользователь переключает режим формы
- **THEN** система очищает пароль, ошибки полей и общую ошибку предыдущей отправки

### Requirement: Адаптивное размещение формы
`main-fe` MUST центрировать auth-контент по горизонтали и вертикали, ограничивать ширину формы на широком экране и использовать доступную ширину с разумными внешними отступами на мобильном экране.

#### Scenario: Широкий экран
- **WHEN** `/login` отображается на широком viewport
- **THEN** форма находится по центру страницы и не превышает заданную максимальную ширину

#### Scenario: Мобильный экран
- **WHEN** `/login` отображается на мобильном viewport
- **THEN** форма использует доступную ширину экрана без горизонтального переполнения и сохраняет внешние отступы

### Requirement: Клиентская валидация форм
`main-fe` MUST валидировать обязательные поля до HTTP-запроса, а регистрационная форма MUST зеркалировать известные ограничения `main-be`: корректный email, логин длиной 3–32 из допустимых символов и пароль длиной 8–128 как минимум с одной буквой и одной цифрой.

#### Scenario: Невалидная регистрационная форма
- **WHEN** пользователь отправляет регистрацию с некорректным email, логином или паролем
- **THEN** система показывает ошибки рядом с соответствующими полями и не отправляет HTTP-запрос

#### Scenario: Пустая форма входа
- **WHEN** пользователь отправляет вход без логина или пароля
- **THEN** система показывает ошибки обязательных полей и не отправляет HTTP-запрос

### Requirement: Регистрация через main-be
`main-fe` MUST отправлять валидную регистрацию только в `main-be` запросом `POST /api/auth/register` с JSON-полями `email`, `username`, `password` и cookie credentials.

#### Scenario: Успешная регистрация
- **WHEN** `main-be` отвечает на регистрацию статусом `201`
- **THEN** система показывает сообщение об успешном создании пользователя, переключается в режим «Вход» и подставляет зарегистрированный логин без автоматического создания сессии

#### Scenario: Конфликт регистрационных данных
- **WHEN** `main-be` отклоняет регистрацию из-за уже используемого email или логина
- **THEN** система сохраняет введённые несекретные значения и показывает понятную серверную ошибку отдельно от ошибок клиентской валидации

#### Scenario: Ошибка validation backend
- **WHEN** `main-be` возвращает validation detail в виде массива
- **THEN** система преобразует ответ в читаемое сообщение и не отображает необработанный объект

### Requirement: Вход через main-be
`main-fe` MUST отправлять валидный вход только в `main-be` запросом `POST /api/auth/token` с JSON-полями `login`, `password`, cookie credentials и MUST считать вход успешным только после ответа `204` и последующей успешной проверки пользователя.

#### Scenario: Успешный вход
- **WHEN** `main-be` отвечает на вход статусом `204`, устанавливает HttpOnly cookie и последующая проверка возвращает пользователя
- **THEN** система обновляет состояние сессии и заменяет маршрут на `/`

#### Scenario: Неверные credentials
- **WHEN** `main-be` отвечает на вход статусом `401`
- **THEN** система остаётся на `/login`, не очищает логин и показывает общую ошибку входа без раскрытия внутренних деталей

#### Scenario: Повторная отправка во время запроса
- **WHEN** запрос входа или регистрации ещё выполняется
- **THEN** система показывает pending-состояние и блокирует повторную отправку формы

### Requirement: Проверка и однократное восстановление сессии
`main-fe` MUST проверять текущего пользователя через `GET /api/auth/verify`; при ответе `401` система MUST выполнить не более одного `POST /api/auth/refresh`, затем не более одного раза повторить verify и MUST объединять параллельные refresh-попытки.

#### Scenario: Действующая access cookie
- **WHEN** verify возвращает данные пользователя
- **THEN** система устанавливает состояние `authenticated` без вызова refresh

#### Scenario: Успешное восстановление
- **WHEN** первый verify возвращает `401`, refresh возвращает `204`, а повторный verify возвращает пользователя
- **THEN** система устанавливает состояние `authenticated`

#### Scenario: Недействительная refresh cookie
- **WHEN** verify и последующий refresh возвращают `401`
- **THEN** система устанавливает состояние `anonymous` без циклических повторов

#### Scenario: Сбой сети при проверке
- **WHEN** verify завершается сетевой или серверной ошибкой, отличной от подтверждённой недействительной сессии
- **THEN** система показывает состояние ошибки с возможностью безопасно повторить проверку и не запускает бесконечное перенаправление

### Requirement: Защита маршрутов
`main-fe` MUST скрывать содержимое защищённого `/` до завершения проверки сессии, перенаправлять подтверждённого гостя на `/login` и перенаправлять подтверждённого аутентифицированного пользователя с `/login` на `/`; отдельный HH OAuth callback-маршрут MUST оставаться доступным без пользовательской сессии и MUST показывать только служебный интерфейс завершения callback.

#### Scenario: Проверка ещё выполняется
- **WHEN** пользователь открывает `/` до завершения verify
- **THEN** система показывает нейтральное loading-состояние и не показывает защищённую заглушку

#### Scenario: Гость открывает главную страницу
- **WHEN** проверка на `/` подтверждает состояние `anonymous`
- **THEN** система заменяет маршрут на `/login`

#### Scenario: Пользователь с сессией открывает login
- **WHEN** проверка на `/login` подтверждает состояние `authenticated`
- **THEN** система заменяет маршрут на `/`

#### Scenario: Пользователь с сессией открывает главную страницу
- **WHEN** проверка на `/` подтверждает состояние `authenticated`
- **THEN** система показывает защищённую главную страницу

#### Scenario: Гость открывает HH OAuth callback
- **WHEN** маршрут callback открыт без действующей cookie-сессии после redirect от HH
- **THEN** система не перенаправляет его на `/login` и показывает только служебное состояние завершения OAuth

### Requirement: Безопасная клиентская граница
`main-fe` MUST получать base URL `main-be` из `NEXT_PUBLIC_API_BASE_URL`, отправлять auth-запросы с `credentials: "include"` и MUST NOT сохранять или логировать пароли, access token, refresh token, email или логин в local storage, session storage, console либо debug UI.

#### Scenario: Сериализация auth-запроса
- **WHEN** feature service вызывает auth endpoint
- **THEN** API client добавляет настроенный base URL, JSON headers и cookie credentials без Authorization header

#### Scenario: Обработка пустого успешного ответа
- **WHEN** `main-be` возвращает успешный `204` без тела
- **THEN** API client обрабатывает ответ без попытки обязательного JSON parsing

#### Scenario: Отсутствует base URL
- **WHEN** `NEXT_PUBLIC_API_BASE_URL` не настроен
- **THEN** система возвращает явную конфигурационную ошибку и не отправляет auth-запрос на угаданный backend origin

### Requirement: DEV-порт main-fe
DEV-команда `main-fe` MUST запускать Next.js на порту `3100`.

#### Scenario: Локальный запуск
- **WHEN** разработчик выполняет `npm run dev` в `services/main-fe`
- **THEN** Next.js запускает development server с параметром порта `3100`

### Requirement: Автономные frontend-тесты
`main-fe` MUST иметь unit/component tests для validators, API boundary, auth orchestration, форм и route gates, которые не требуют live backend.

#### Scenario: Запуск тестов без main-be
- **WHEN** разработчик запускает frontend test script при остановленном `main-be`
- **THEN** тесты используют подменённую сетевую границу и проверяют success, validation, loading, backend/network error и навигацию
