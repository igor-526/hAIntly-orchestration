## Purpose

Определить read-only HTTP-контракты доступа к активным справочникам вакансий через `vacancy-service` и авторизованный proxy `main-be`.

## Requirements

### Requirement: Read-only API активных справочников
`vacancy-service` MUST предоставлять внутренние LIST и GET операции для элементов справочников полей, регионов, стран, профессиональных ролей, отраслей, городов/линий/станций метро и языков. API MUST возвращать только `active=true` по умолчанию и MUST NOT предоставлять POST, PATCH или DELETE операции.

#### Scenario: Получение списка
- **WHEN** внутренний клиент запрашивает список допустимого семейства справочника
- **THEN** сервис возвращает пагинированные активные записи с их HH ID, отображаемыми полями и применимыми родительскими идентификаторами

#### Scenario: Получение активной записи
- **WHEN** внутренний клиент запрашивает существующий активный HH ID в однозначном пространстве справочника
- **THEN** сервис возвращает одну запись этого справочника

#### Scenario: Запись отсутствует или неактивна
- **WHEN** GET указывает неизвестный либо неактивный HH ID
- **THEN** сервис возвращает `404` без обращения к HH в рамках запроса

#### Scenario: Попытка изменения
- **WHEN** клиент отправляет POST, PATCH или DELETE в namespace справочников
- **THEN** операция не маршрутизируется как поддерживаемый endpoint и данные не изменяются

### Requirement: Ограниченный регистронезависимый текстовый поиск
Каждая применимая LIST операция MUST поддерживать параметр `q`, который сопоставляется с отображаемым названием без учёта регистра. Пользовательское значение MUST интерпретироваться как буквальный текст, иметь документированный предел длины и не должно исполняться как произвольное регулярное выражение; результаты MUST оставаться пагинированными.

#### Scenario: Поиск с другим регистром
- **WHEN** `q` содержит часть названия в регистре, отличном от сохранённого
- **THEN** список содержит соответствующую активную запись

#### Scenario: Метасимволы регулярного выражения
- **WHEN** `q` содержит символы регулярных выражений
- **THEN** сервис экранирует их и ищет буквальную последовательность без расширения шаблона

#### Scenario: Слишком длинный запрос
- **WHEN** длина `q` превышает документированный предел
- **THEN** сервис возвращает ошибку валидации и не выполняет запрос к БД

### Requirement: Авторизованный proxy через main-be
`main-be` MUST предоставлять авторизованному браузерному пользователю зеркальные read-only LIST/GET операции и MUST получать данные только по HTTP из `vacancy-service`. Адрес, timeout и применимый исходящий service credential MUST поступать из валидируемого окружения без production hardcode; если отдельная входящая service-auth у `vacancy-service` не предусмотрена общей политикой закрытой сети, `main-be` MUST NOT изобретать новый credential в рамках этого change.

#### Scenario: Авторизованный proxy-запрос
- **WHEN** пользователь с действующей access cookie запрашивает справочник через `main-be`
- **THEN** `main-be` вызывает `vacancy-service`, сохраняет параметры пагинации/поиска и возвращает стабильный публичный ответ

#### Scenario: Неавторизованный публичный запрос
- **WHEN** браузерный запрос не содержит действующей access cookie
- **THEN** `main-be` возвращает `401` и не вызывает `vacancy-service`

#### Scenario: Конфигурация зависимости отсутствует
- **WHEN** обязательный адрес `vacancy-service`, timeout или требуемый общей политикой credential отсутствует либо невалиден
- **THEN** `main-be` завершает загрузку настроек конфигурационной ошибкой до обслуживания запросов

### Requirement: Пользовательский контекст внутреннего HTTP-запроса
Для proxy-вызова `main-be` MUST брать UUID из аутентифицированного пользователя и передавать его в `X-User-Id`. `vacancy-service` MUST валидировать наличие и формат UUID, MUST NOT использовать его для ограничения общедоступных данных и MUST NOT сохранять его в таблицах справочников.

#### Scenario: Валидный пользовательский заголовок
- **WHEN** `main-be` передаёт UUID аутентифицированного пользователя в `X-User-Id`
- **THEN** `vacancy-service` обслуживает read-only запрос независимо от конкретного пользователя

#### Scenario: Заголовок отсутствует
- **WHEN** внутренний запрос не содержит `X-User-Id`
- **THEN** `vacancy-service` возвращает `400` и не выполняет чтение справочника

#### Scenario: Заголовок не является UUID
- **WHEN** `X-User-Id` содержит невалидное значение
- **THEN** `vacancy-service` возвращает `400` без раскрытия внутренних деталей

### Requirement: Стабильная обработка ошибок зависимости
`main-be` MUST ограничивать время внутреннего HTTP-вызова и преобразовывать timeout, недоступность и неожиданный ответ `vacancy-service` в стабильный gateway-ответ без утечки внутренних адресов, credentials, traceback или полного тела внешней ошибки.

#### Scenario: Vacancy-service недоступен
- **WHEN** внутренний вызов завершается timeout или ошибкой соединения
- **THEN** `main-be` возвращает документированную временную gateway-ошибку и безопасно логирует класс сбоя

#### Scenario: Клиентская ошибка параметров
- **WHEN** `vacancy-service` отклоняет невалидные параметры пагинации или поиска
- **THEN** `main-be` возвращает соответствующую стабильную ошибку валидации без маскировки под успешный ответ
