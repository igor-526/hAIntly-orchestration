## Purpose

Определить единый воспроизводимый интерфейс форматирования, статического анализа и тестирования для всех проектов HAIntly, а также обязательные проверки профильных исполнителей и Quality Gate.

## Requirements

### Requirement: Единый Makefile-интерфейс качества
Каждый из проектов `main-be`, `profile-service`, `vacancy-service`, `ai-service`, `main-fe` и `fastapi_template` MUST предоставлять исполняемые цели `make format`, `make lint` и `make test`, каждая из которых запускает полный применимый автономный pipeline соответствующего проекта и возвращает ненулевой код при ошибке. `make test` MUST выполняться после установки зависимостей текущего проекта без Docker, внешней БД, Redis, NATS, соседнего checkout HAIntly-сервиса, `.venv` другого проекта, внешнего провайдера или production secrets.

#### Scenario: Успешный автономный pipeline сервиса
- **WHEN** разработчик или GitHub Actions последовательно запускает `make format`, `make lint` и `make test` из каталога проекта с установленными зависимостями текущего проекта и без запущенной HAIntly-инфраструктуры
- **THEN** каждая цель выполняет реальную проверку или форматирование и завершается успешно

#### Scenario: Ошибка автономного инструмента качества
- **WHEN** formatter, linter, type checker или автономный тестовый runner внутри соответствующей цели завершается с ошибкой
- **THEN** Makefile-цель также завершается с ненулевым кодом и не маскирует ошибку

#### Scenario: Одинаковый набор локально и в CI
- **WHEN** `make test` запускается для одной ревизии и набора зависимостей локально и в GitHub Actions
- **THEN** команда выбирает один и тот же автономный набор тестов без ветвления по `CI` или `GITHUB_ACTIONS`

### Requirement: Воспроизводимое frontend-форматирование
`main-fe` MUST иметь formatter, зафиксированный в development-зависимостях и lock-файле, а `make format` MUST применять его к поддерживаемым frontend-файлам без зависимости от глобально установленного инструмента.

#### Scenario: Форматирование frontend-файла
- **WHEN** поддерживаемый файл `main-fe` не соответствует правилам formatter и выполняется `make format`
- **THEN** файл приводится к установленному формату локально зафиксированной версией инструмента

#### Scenario: Повторный запуск frontend format
- **WHEN** `make format` повторно выполняется без иных изменений после успешного форматирования
- **THEN** команда не создаёт новый diff

### Requirement: Защита инфраструктурных каталогов от форматирования
Formatter каждого затронутого проекта MUST исключать любой каталог `.github` и `.helm` и всё их содержимое из изменяющей операции `make format`, включая каталоги, созданные после внедрения change.

#### Scenario: Существующие инфраструктурные каталоги
- **WHEN** `.github` или `.helm` содержит файл, который formatter иначе изменил бы, и запускается `make format`
- **THEN** содержимое файла остаётся без изменений

#### Scenario: Каталог добавлен после настройки formatter
- **WHEN** каталог `.github` или `.helm` создаётся позднее в области проекта и затем запускается `make format`
- **THEN** конфигурация исключения применяется без дополнительного изменения Makefile или formatter-конфигурации

### Requirement: Качественное состояние затронутых проектов
Реализация MUST устранять ошибки format, lint и автономного test pipeline в затронутых проектах без отключения проверок, ослабления правил либо исключения product/test исходников ради зелёного результата. Тесты MAY быть исключены из `make test` только при фактической зависимости от инфраструктуры и MUST оставаться доступными через полный локальный infrastructure pipeline.

#### Scenario: Проверка после исправлений
- **WHEN** полный набор из `make format`, `make lint` и `make test` запускается в каждом затронутом проекте без внешней инфраструктуры
- **THEN** все команды завершаются успешно, а исправления не меняют заявленные продуктовые контракты и поведение

#### Scenario: Infrastructure-тест исключён из автономного набора
- **WHEN** тест требует Docker, внешнюю БД, Redis, NATS, другой HAIntly-сервис, внешний провайдер или production secret
- **THEN** тест явно классифицирован как infrastructure-тест, не выполняется командой `make test` и выполняется отдельной локальной infrastructure-командой

#### Scenario: Для зелёного результата требуется продуктовый change
- **WHEN** обнаруженная ошибка не может быть исправлена без изменения продуктового поведения, API, события, БД или сервисной границы
- **THEN** реализация останавливается до явного обновления OpenSpec вместо обхода проверки

### Requirement: Явная классификация infrastructure-тестов
Backend-проект с тестами, зависящими от локальной инфраструктуры или другого HAIntly-сервиса, MUST регистрировать pytest marker `infrastructure`, применять его ко всем таким тестам и включать strict-проверку неизвестных markers. Каталог или имя `integration` сами по себе MUST NOT исключать автономный тест из CI.

#### Scenario: Тест требует реальную инфраструктуру
- **WHEN** pytest-тест обращается к реальному PostgreSQL, Redis, NATS, Docker, миграциям реальной БД, внешнему провайдеру или процессу другого HAIntly-сервиса
- **THEN** тест помечен marker `infrastructure`

#### Scenario: Integration-тест автономен
- **WHEN** integration-тест использует только in-process приложение, mock/fake зависимости, временные файлы или локальный loopback-сервер текущего проекта
- **THEN** тест остаётся в наборе `make test` и не исключается только из-за расположения в каталоге `integration`

#### Scenario: Marker указан с ошибкой
- **WHEN** тест содержит незарегистрированный pytest marker
- **THEN** сбор или запуск тестов завершается с ненулевым кодом

### Requirement: Полный локальный infrastructure pipeline
Каждый backend-проект MUST предоставлять цель `make test-infra`. При наличии тестов с marker `infrastructure` цель MUST выполнять все такие тесты, использовать адреса и credentials только из локального окружения и до запуска выдавать конкретную диагностическую ошибку при отсутствии обязательной зависимости. Команда MUST NOT молча пропускать infrastructure-тесты из-за недоступности зависимости.

#### Scenario: Локальная инфраструктура доступна
- **WHEN** разработчик поднял требуемые контейнеры или сервисы, настроил локальное окружение и запускает `make test-infra`
- **THEN** выполняются все тесты с marker `infrastructure`, а ошибка любого из них возвращает ненулевой код

#### Scenario: Локальная инфраструктура отсутствует
- **WHEN** обязательная БД, Redis, Docker-контейнер, соседний HAIntly-сервис или его исполняемое окружение недоступны
- **THEN** `make test-infra` завершается с ненулевым кодом и сообщает, какую зависимость и каким предусмотренным проектом способом необходимо подготовить

#### Scenario: Секреты локальной проверки
- **WHEN** infrastructure-команда загружает credentials или service credential
- **THEN** значения берутся из окружения или локального untracked env-файла и не выводятся в консоль, GitHub Actions или Quality Gate отчёт

#### Scenario: В сервисе пока нет infrastructure-тестов
- **WHEN** backend-проект не содержит тестов с marker `infrastructure`
- **THEN** проект всё равно предоставляет исполняемую цель `make test-infra`, которая успешно сообщает об отсутствии применимых infrastructure-тестов и не создаёт фиктивные тесты

### Requirement: Наследуемый тестовый контракт backend-сервисов
`fastapi_template` MUST содержать автономный `make test`, зарегистрированный strict pytest marker `infrastructure` и исполняемую цель `make test-infra`. Каждый новый backend-микросервис MUST создаваться с этим контрактом, а каждый добавляемый или изменяемый pytest-тест MUST быть классифицирован в момент разработки по фактическим runtime-зависимостям. Quality Gate MUST независимо проверять классификацию.

#### Scenario: Создаётся новый backend-микросервис
- **WHEN** разработчик копирует `fastapi_template` для нового FastAPI-сервиса
- **THEN** новый сервис сразу предоставляет автономный `make test`, strict marker `infrastructure` и цель `make test-infra`

#### Scenario: Добавляется автономный тест
- **WHEN** новый или изменённый pytest-тест использует только in-process приложение, mock/fake зависимости, временные ресурсы или локальный loopback текущего проекта
- **THEN** тест не получает marker `infrastructure` и выполняется командой `make test`

#### Scenario: Добавляется тест с внешней runtime-зависимостью
- **WHEN** новый или изменённый pytest-тест требует реальную инфраструктуру, внешний процесс, другой HAIntly-сервис, внешний провайдер или secret
- **THEN** разработчик при создании теста применяет marker `infrastructure`, а тест выполняется через `make test-infra` и не входит в `make test`

#### Scenario: Quality Gate проверяет классификацию
- **WHEN** change добавляет или изменяет backend-тесты
- **THEN** Quality Gate сопоставляет зависимости каждого такого теста с marker, запускает автономный набор без инфраструктуры и применимый infrastructure-набор отдельно

### Requirement: Автономность GitHub Actions отдельных сервисов
GitHub Actions каждого реализованного сервиса MUST выполнять `make test` в checkout только этого сервиса и MUST завершаться без попытки запуска инфраструктурных тестов. Workflow MUST NOT поднимать service containers только для удовлетворения infrastructure-набора этого change.

#### Scenario: Проверка main-be в отдельном checkout
- **WHEN** workflow `main-be` запускает `make test` без соседнего каталога `profile-service`
- **THEN** автономные тесты выполняются, а межсервисный smoke с реальным `profile-service` не запускается

#### Scenario: Проверка vacancy-service без Docker-инфраструктуры
- **WHEN** workflow `vacancy-service` запускает `make test` без PostgreSQL и Redis HAIntly
- **THEN** автономные API/unit/in-process тесты выполняются, а PostgreSQL-, Redis- и migration-тесты не запускаются

#### Scenario: Автономный тест падает
- **WHEN** любой тест, входящий в автономный набор сервиса, завершается ошибкой в GitHub Actions
- **THEN** `make test` и job workflow завершаются с ненулевым кодом

### Requirement: Обязательная проверка профильным исполнителем
Профили Backend Agent и Frontend Agent MUST требовать непосредственно перед передачей запуск `make format`, `make lint` и `make test` в каждом затронутом ими сервисе; отсутствие цели, ошибка или пропуск любой команды MUST блокировать готовность к Quality Gate.

#### Scenario: Успешная передача реализации
- **WHEN** профильный агент завершил изменения и все три команды успешно выполнены в каждом затронутом им сервисе
- **THEN** агент фиксирует результаты команд в передаче и может объявить реализацию готовой к Quality Gate при выполнении остальных профильных проверок

#### Scenario: Команда не прошла
- **WHEN** хотя бы одна обязательная Makefile-команда отсутствует, пропущена или завершилась ошибкой
- **THEN** профильный агент не передаёт работу как готовую к Quality Gate

### Requirement: Независимая проверка Quality Gate
Профиль Quality Gate MUST независимо запускать `make format`, `make lint` и `make test` в каждом сервисе, затронутом change, и MUST считать отсутствие, пропуск или ошибку любой команды основанием для решения `НА ДОРАБОТКУ`.

#### Scenario: Все сервисы прошли независимую проверку
- **WHEN** Quality Gate выполнил три обязательные команды в каждом затронутом сервисе и все они успешны
- **THEN** результаты включаются в отчёт как доказательство и review продолжается по остальным обязательным критериям

#### Scenario: Один сервис не прошёл матрицу
- **WHEN** хотя бы в одном затронутом сервисе обязательная команда отсутствует, пропущена или завершается ошибкой
- **THEN** Quality Gate возвращает `НА ДОРАБОТКУ` и указывает сервис, команду и наблюдаемый результат

### Requirement: Автономная конфигурация стадии pytest collection
Backend-проект MUST обеспечивать import и collection автономного pytest-набора без service `.env`, production secrets и обязательных runtime-настроек внешних интеграций. Если application-модули создают валидируемую конфигурацию при импорте, test harness MUST до такого импорта задавать явные безопасные test-only значения, MUST сохранять приоритет уже заданного окружения и MUST NOT ослаблять обязательность runtime-конфигурации вне тестов.

#### Scenario: Collection в отдельном checkout без runtime environment
- **WHEN** `vacancy-service` запускает `make test` в checkout без сервисного `.env` и без переменных HH, Celery, Redis, словарей и `profile-service`
- **THEN** pytest успешно импортирует `conftest.py` и application-модули, собирает автономный набор и не завершается ошибкой валидации `Settings`

#### Scenario: Безопасные значения не инициируют интеграции
- **WHEN** test harness задаёт фиктивные URL, DSN, token и user-agent для построения конфигурации автономных тестов
- **THEN** автономный набор не выполняет реальных обращений к HH, PostgreSQL, Redis, Celery broker/backend или `profile-service`

#### Scenario: Infrastructure environment имеет приоритет
- **WHEN** `make test-infra` заранее загрузил реальные локальные адреса и credentials из разрешённого окружения или untracked env-файла
- **THEN** test harness сохраняет эти значения и infrastructure-тесты используют подготовленную локальную инфраструктуру

#### Scenario: Runtime-запуск без обязательной конфигурации
- **WHEN** приложение или фоновый процесс `vacancy-service` запускается вне test harness без обязательного параметра внешней интеграции
- **THEN** `Settings` завершает старт с явной ошибкой валидации, не подставляя test-only значение или production default

### Requirement: Регрессия автономного запуска проверяется без локального env-файла
Изменение test configuration MUST включать автоматизированную проверку, которая воспроизводит import/collection автономного набора в очищенном окружении без использования tracked или локального service `.env`. Проверка MUST возвращать ненулевой код при повторном появлении зависимости collection от обязательной runtime-конфигурации.

#### Scenario: Локальный env-файл маскирует отсутствие переменных
- **WHEN** на рабочей машине присутствует `vacancy-service/.env`, но регрессионная проверка запускается
- **THEN** проверка изолирует subprocess или test probe от этого файла и подтверждает самодостаточность test harness

#### Scenario: Обязательный test default удалён
- **WHEN** application import снова требует runtime-поле, которое test harness не предоставляет в очищенном окружении
- **THEN** регрессионная проверка завершается с ненулевым кодом до признания `make test` совместимым с GitHub Actions
