# Архитектура агентов HAIntly

HAIntly — микросервисный AI-инструмент для поиска релевантных вакансий HeadHunter, их оценки относительно резюме и подготовки сопроводительных писем.

В проекте используется OpenSpec-first процесс и мультиагентная маршрутизация. Смысловое содержание ответов, отчётов и OpenSpec-артефактов пишется на русском языке. Служебные маркеры OpenSpec, необходимые CLI для разбора спецификаций, сохраняются на английском языке.

## Источники правды

- Активный `openspec/changes/<change>/` — требования, дизайн и задачи конкретного изменения.
- `openspec/specs/` — действующие требования после синхронизации change.
- `SERVICES.md` — утверждённые границы сервисов и способы взаимодействия.
- `docs/information.md` — краткое продуктовое описание HAIntly.
- `agents/*.md` — правила профильных исполнителей.

Если документы расходятся, реализация останавливается до уточнения или обновления OpenSpec. Markdown-планы вне OpenSpec не являются источником планирования.

## Роль Router

Router принимает запрос, определяет процесс и запускает профильных subagent. Router не пишет код и не выполняет Quality Gate вместо профильного агента.

Для каждого запроса Router:

1. Определяет режим: исследование, планирование, backend, frontend или Quality Gate.
2. Проверяет наличие подходящего OpenSpec change.
3. Запускает нужный OpenSpec skill или профильного subagent.
4. Передаёт исполнителю профиль `agents/*.md`, OpenSpec context и связанные файлы.
5. Требует продолжать работу до завершения, останавливаясь только при конкретном блокере.
6. После реализации обязательно запускает отдельный Quality Gate.
7. Возвращает пользователю итог на русском языке.

Если среда не предоставляет инструмент запуска subagent, Router не подменяет исполнителя, а сообщает о техническом блокере.

## OpenSpec-процесс

### Исследование

Для анализа идеи без изменений используется `openspec-explore`. Исследование не создаёт код и не считается реализацией.

### Propose

Новая, крупная, архитектурная или межсервисная задача сначала оформляется через `openspec-propose`. Входом может быть текст пользователя или `docs/prompts/<slug>.md`.

Change должен содержать:

- `proposal.md` — цель, пользовательская ценность, scope и затронутые сервисы;
- `design.md` — решения, границы данных и интеграционные потоки;
- `tasks.md` — атомарный маршрутизируемый checklist;
- `specs/*` — проверяемые требования и сценарии.

Перед `apply` пользователь проверяет артефакты. Неясное или противоречивое ТЗ не передаётся в реализацию.

### Apply

Для существующего change Router вызывает `openspec-apply-change`, читает все `contextFiles` из CLI и распределяет незавершённые задачи:

- `Backend` → Backend Agent;
- `Frontend` → Frontend Agent;
- `Quality Gate` → Quality Gate Agent только после завершения реализации.

Параллельно можно выполнять только независимые задачи с непересекающимися файлами. Каждый исполнитель сразу отмечает завершённые пункты в `tasks.md`. Если реализация расходится с design/specs, сначала обновляются OpenSpec-артефакты.

Если незавершённых задач много, Router распределяет их между несколькими профильными subagent только при независимых областях ответственности и непересекающихся файлах. Каждому исполнителю передаются только его профиль, обязательные OpenSpec context files, связанные файлы и зависимости его задач; перегружать subagent контекстом всего change без необходимости запрещено.

### Quality Gate и завершение

Quality Gate запускается отдельным subagent и возвращает `ОДОБРЕНО` или `НА ДОРАБОТКУ`. При доработке замечания превращаются в атомарные задачи соответствующих секций, после чего `apply` и Quality Gate повторяются.

После одобрения delta specs синхронизируются через `openspec-sync-specs`, а change архивируется через `openspec-archive-change`.

### Bootstrap-исключение

Прямое изменение `AGENTS.md`, `agents/*.md`, `openspec/config.yaml` и документов, описывающих сам workflow, допускается только при первоначальной настройке или ремонте оркестратора. Исключение не распространяется на продуктовый код, архитектурные изменения сервисов и пользовательское поведение.

## Протокол нового backend-сервиса

- Новый FastAPI-сервис создаётся копированием `fastapi_template` в `services/<service-name>`, после чего шаблонные имена и настройки адаптируются под сервис.
- Каждый сервис получает отдельный `.docker-compose/docker-compose.<service>.yml` с приложением и миграциями, но без PostgreSQL или другой общей инфраструктуры. БД и инфраструктура поднимаются только через `.docker-compose/docker-compose.infra.yml` и могут находиться на другом хосте.
- Compose project name микросервиса использует шаблон `haintly-<service-name>`, например `haintly-profile-service`.
- Корневой `Makefile` предоставляет `<service>` для запуска compose и `<service>-build` только для сборки образов без запуска. Makefile каждого сервиса обязательно наследует из `fastapi_template` цели `lint`, `format`, автономную `test` и отдельную `test-infra`.
- `make test` backend-сервиса не требует Docker, HAIntly infrastructure, соседнего checkout, внешнего провайдера или secrets. Pytest-тест с такой runtime-зависимостью получает strict marker `infrastructure` и запускается только через `make test-infra`.
- Каждый новый или изменяемый backend-тест классифицируется по фактическим зависимостям при создании: автономный тест остаётся в `make test`, infrastructure-тест сразу получает marker `infrastructure`. Наличие каталога `integration` само по себе не определяет классификацию.
- Все сервисы публикуют требуемые локальные порты. Индекс компонента совпадает в диапазонах FastAPI `81xx`, PostgreSQL `54xx` и frontend `31xx`: main — `01`, profile — `02`, vacancy — `03`, AI — `04`, notification — `05`.
- Credentials PostgreSQL хранятся в `.docker-compose/.env` и называются `<SERVICE>_POSTGRES_USER`, `<SERVICE>_POSTGRES_PASSWORD`, `<SERVICE>_POSTGRES_DB`, `<SERVICE>_POSTGRES_EXP_PORT`; значения используют `<service>_user`, `<service>_password`, `<service>_db` и порт `54xx` соответствующего индекса. Tracked example-файлы содержат только безопасные placeholders.
- Микросервисы доступны только во внутренней сети и не реализуют пользовательскую или отдельную входящую авторизацию без явного требования OpenSpec. Публичной точкой входа остаётся `main-be`.
- Для пользовательского контекста во внутренних HTTP-вызовах используется заголовок `X-User-Id`. Вызов защищённого пользовательского endpoint `main-be` от микросервиса использует согласованный OpenSpec service credential вместо cookie.
- Адрес каждой исходящей HTTP-зависимости, service credential, NATS URL/subjects/stream, Redis, Celery и другие связанные параметры задаются окружением вызывающего сервиса и валидируются при старте. Hardcode адресов и инфраструктурных контрактов в production-коде или compose запрещён.
- `profile-service` и архивный change `task-04-hh-oauth` используются как референс структуры, контейнеризации и интеграционных границ, но не отменяют требования текущего OpenSpec change.

## Протокол HeadHunter API

При любой работе с HeadHunter исполнитель обязан сверить локальную OpenAPI-спецификацию `docs/hh_openapi.yml` и актуальную документацию `https://api.hh.ru/openapi/redoc`. Endpoint, параметры, ограничения и обработка ошибок не выводятся только из памяти или существующего кода.

## Runtime-проверки реализации

- Backend или Frontend Agent не завершает передачу, пока изменённые контейнеры не собраны заново при необходимости, не подняты и не прошли health/runtime-проверку.
- Quality Gate обязательно выполняет применимые тесты и изучает логи связанных контейнеров. Для диагностики разрешено подключение к тестовой БД по credentials из `.docker-compose/.env`; восстанавливать тестовые данные после проверки не требуется.
- Для изменённых backend endpoint выполняется smoke напрямую к API профильного сервиса и через `main-be` как авторизованный proxy. Для cookie-auth используются `admin@admin.ru` / `user@user.ru` с тестовым паролем из локального задания; секреты, cookie и service credential не попадают в отчёт.
- Изменения фоновых процессов требуют e2e: инициировать операцию, дождаться результата и проверить ожидаемый внешний эффект или записи в БД.
- После успешного Quality Gate reviewer создаёт отчёт в `docs/reports` на основе `docs/reports/TEMPLATE.md`.

## Карта агентов

| Агент | Профиль | Ответственность |
|---|---|---|
| Backend | `agents/backend.md` | Python/FastAPI, БД, интеграции, очереди и backend-тесты |
| Frontend | `agents/frontend.md` | Next.js, UI, HTTP/SSE boundary и frontend-тесты |
| Quality Gate | `agents/quality_gate.md` | Проверка diff, OpenSpec, архитектуры, безопасности и тестов |

## Карта сервисов

| Сервис | Владелец ответственности |
|---|---|
| `main-fe` | Пользовательский интерфейс и получение уведомлений по SSE |
| `main-be` | Пользователи, auth, API gateway, оркестрация и SSE |
| `profile-service` | HH OAuth, связи аккаунтов, защищённые токены и резюме |
| `vacancy-service` | Сбор вакансий, пользовательские статусы и запуск AI-обработки |
| `ai-service` | Оценка релевантности, письма, AI-настройки и статистика токенов |
| `notification-service` | История, настройки и доставка пользовательских уведомлений |

Сервис владеет своей БД. Прямой доступ к БД другого сервиса запрещён. Изменение границ, контрактов HTTP/SSE, событий NATS JetStream или владения данными требует OpenSpec-решения.

## Формат `tasks.md`

OpenSpec требует нумерованные заголовки и checkbox-задачи. Используется структура:

```markdown
## 1. Backend

### main-be
- [ ] 1.1 Проверяемая задача

### profile-service
- [ ] 1.2 Проверяемая задача

## 2. Frontend

### main-fe
- [ ] 2.1 Проверяемая задача

## 3. Quality Gate

- [ ] 3.1 Проверить соответствие change и выполнить доступные проверки
```

Ненужные подразделы не добавляются. Для любого изменения кода секция Quality Gate обязательна.

## Протокол передачи

```text
Задача: <краткое описание>
Область: <backend | frontend | quality-gate>
Сервис: <техническое имя или несколько имён>
OpenSpec change: <имя>
Артефакты: <пути из OpenSpec contextFiles>
Связанные файлы: <ключевые файлы>
Ограничения: <границы, контракты, язык, запреты>
Ожидаемый результат: <код, тесты, checklist или review>
Инструкция: прочитай профиль агента; продолжай до завершения и не жди дополнительных инструкций без конкретного блокера.
```

## Чеклист Router

- Выбран правильный OpenSpec-процесс или профиль.
- Для продуктовой реализации существует подходящий change.
- Исполнителю переданы профиль, артефакты и сервисные границы.
- Router не выполнил профильную работу самостоятельно.
- После изменения кода выполнен отдельный Quality Gate.
- Итог дан на русском языке.
