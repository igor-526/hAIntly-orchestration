В задачах 07 и 08 были реализованы системы хранения справочников и хранения/управления фильтрами. В этой итерации мы будем реализовывать систему выдачи вакансий
# Доступ
Выбранный профиль HeadHunter хранится в Main Backend
Токены в зашифрованном виде хранятся в Profile Service
Для получения списка ваканский нам необходимо обратиться к MainBackend за выбранным аккаунтом и потом к Profile Service за access токеном этого аккаунта
## Передача токена
Profile Service хранит токен в зашифрованном виде. Нам нужно реализовать сервисный GET метод, который будет возвращать актуальный access токен уже в незашифрованном виде. Этот метод будет использоваться только другими сервисами и не должен быть публичным. Только прямое обращение по HTTP к profile-service
## refresh
Access токен может протухнуть. Для этого необходимо реализовать систему refresh токенов. Я предлагаю это сделать в том же запросе, что и на access токен, только с передачей дополнительного параметра refresh=true. Если refresh по каким-либо причинам невозможен, то в таком случае удаляется аккаунт HeadHunter
# Новый функционал vacancy-service
Нам теперь нужно иметь возможность получать как список вакансий согласно фильтрам, так и отображать одну вакансию. 
## Получение списка вакансий
Метод на получение вакансий в сервисе `vacancy-service` должен иметь возможность принимать все доступные (кроме deprecated) параметры фильтрации и пресет фильтров. Все поля nullable. Обрати внимание на явно переданный null и отсутстствие значения. Это могут быть разные вещи. Обрабатываем следующие ситуации:
### Case: все null
Выполняем запрос к HH на получение вакансий без фильтрации
### Case: некоторые или все поля кроме пресета не null
Выполняем запрос к HH на получение вакансий с переданными параметрами фильтрации
### Case: передан только id пресета
Получаем по id пресет и получаем параметры фильтрации, затем выполняем запрос к HH на получение вакансий с полученными из пресета параметрами фильтрации
### Case: передан id пресета и некоторые или все параметры фильтрации
Получаем по id пресет и получаем параметры фильтрации. Переданные кроме пресета параметры переопределяем без сохранения и затем выполняем запрос к HH на получение вакансий с полученными из пресета параметрами фильтрации с переопределением
## Получение одной вакансии
По id вакансии необходимо получать все данные этой вакансии

# Доработки main-fe
Мы реализовали UX, в котором рабочая область делится на две половины. В левой список вакансий, в правой область работы с вакансией. При первичной загрузке нам необходимо получить список вакансий без применения фильтрации. В правой области текст о том, что для начала работы с вакансией необходимо выбрать хотя бы одну из списка
При нажатии на вакансию из списка справа должна открываться полная информация об этой вакансии.
В списке отобрази только необходимые сведения о вакансии, справа всю доступную информацию. Сделай так, как посчитаешь логичным. Сейчас реализуем только отображение, без работы с вакансией
Учти, что в будущем будет необходима возможность настройки отображения как левого списка, так и отображения блоков справа (что отображать, что нет. Сейчас это не реализовываем)
