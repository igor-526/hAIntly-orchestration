# Контекст
Нам нужно запротоколировать некоторое поведение агентов. Используй bootstrap без оркестрации агентами. Нам необходимо изменить файлы агентов, конфигурации OpenSpec и другие файлы, влияющие на поведение агентов для приведение их поведения в соответствие с нижеописанными правилами
## Протокол нового микросервиса
### Шаблон
В папке `fastapi_template` есть шаблон пустого микросервиса без авторизации и прочего
Если мы создаём новый микросервис, то нам необходимо скопировать этот шаблон в целевой сервис папки `services`
Кстати, там есть middleware CORS. Давай уберём его из `fastapi_template` и `profile_service`, так как HTTP запросы к микросервисам может слать только `main_be` и при деплое все микросервисы будут только в локальной сети
### Контейнеризация
В папке `.docker-compose` есть файлы, с помощью которых поднимаются все сервисы:
- `.docker-compose/docker-compose.be.yml` - основной Backend и миграции БД. Все микросервисы должны быть под name `haintly-service`
- `.docker-compose/docker-compose.fe.yml` - frontend
- `.docker-compose/docker-compose.infra.yml` - инфраструктура (всё обслуживающее и БД)
- `.docker-compose/docker-compose.profile.yml` - Backend и миграции БД для микросервиса profile-service. 
Каждый микросервис должен иметь отдельный docker-compose файл без поднятия БД внутри, так как БД может быть вообще на другом хосте
### Database credentials
Так как БД поднимается в инфраструктурном docker-compose, то и файл .env находится в папке с этими файлами.
Именование переменных и их значения должны происходить по следующему шаблону:
```plaintext
<SERVICE>_POSTGRES_USER=<service>_user
<SERVICE>_POSTGRES_PASSWORD=<service>_password
<SERVICE>_POSTGRES_DB=<service>_db
<SERVICE>_POSTGRES_EXP_PORT=5451
```
Например, для profile_service `PROFILE_POSTGRES_USER=profile_user`
### Makefile оркестратора
В корневом Makefile должны быть следующие команды:
- `<service>` (be, fe, profile)- запускает docker compose выбранного сервиса
- `<service>-build` (be-build, fe-build, profile-build) - собирает образы выбранного сервиса с нуля
Build команда не должна поднимать сервис
### Makefile сервиса
Makefile любого сервиса должен иметь команды `lint`, `test` и `format`.
`lint` должен проводить проверку на качество кода
`format` должен форматировать весь код сервиса
`test` должен запускать весь пайплайн тестирования сервиса
### Порты
Все сервисы должны пробрасывать наружу порты. Последняя цифра порта для сервиса, БД и тд должна быть одинаковая. Для FastAPI это 81xx, для PostgreSQL это 54xx, для frontend это 31xx
Например, main-be. API 81**01**, БД 54**01**, фронт 31**01**
profile-service. API 8102, БД 5452, фронт не требуется
### Авторизация
Авторизация микросервисов не требуется, так как микросервисы будут работать исключительно в локальной сети и к ним будет обращаться только основной Backend
### Авторизация основного Backend
При общении микросервисы могут слать запросы к основному Backend без авторизации
Давай реализуем такую авторизацию для микросервисов на основном Backend
Для этого на основном Backend сделаем сервисный ключ. Микросервисы будут передавать его в заголовках и это будет заменять авторизацию по Cookies. Для идентификации пользователя будем использовать заголовок `X-User-Id` как на стороне основного backend, так и на стороне микросервисов
### Настраиваемость адресов сервисов
Основной backend может общаться с микросервисами
Микросервисы могут общаться с основным backend
Микросервисы могут общаться с другими микросервисами
Настройка адресов других сервисов, которые необходимы для работы сервису необходимо проводить через переменную окружения
Например, если `main-backend` делает HTTP запросы к `profile-service`, то в `.env` основного backend должен быть адрес `profle-service`
Если `profle-service` делает HTTP запросы к `main-backend`, то в `.env` `profle-service`должен быть адрес и сервисный ключ `main-be`
Если сервисы общаются друг с другом через NATS, то в обоих сервисах должно быть конфигурирование как адреса NATS, так и других связанных объектов RPC
Аналогичное должно происходить и с Redis, Celery и любой другой инфраструктурой
Хардкод не допускается

## Протокол оркестрации агентами через apply
Иногда задач может быть слишком много. В таком случае не стоит их доверять одному агенту, а раскидать задачи так, чтобы они были в непересекающихся областях и агенты, которые их выполняют не мешали друг другу. Используй доступный контекст субагентов разумно, не перегружай

## Протокол работы QualityGate и других агентов
### Поднятые контейнеры
Backend или frontend агент не может завершить работу, если контейнеры микросервиса не подняты или не пересобраны в случае изменений. Часто бывают ситуации, что QG пройден, а контейнер не поднимается из-за банальных ошибок. Эти ситуации нужно исключить
### Тестирование
QualityGate должен обязательно выполнить тестирование
В случае доработок Backend (в том числе микросервисов) необходимо проводить SMOKE тестирование новых или изменённых эндпоинтов
Проводить SMOKE тестирование необходимо как к API микросервиса, так и используя основной Backend в качестве прокси с авторизацией. Пользователи для авторизации для SMOKE тестирования:

| Логин          | Пароль      | Комментарий                   |
| -------------- | ----------- | ----------------------------- |
| admin@admin.ru | "string123" | Аккаунт администратора        |
| user@user.ru   | "string123" | Аккаунт обычного пользователя |
### Инструменты тестирования
При тестировании необходимо обращаться к логам связанных контейнеров
При тестировании допустимо использование подключения к БД для анализа записей. Для этого необходимо найти контейнер БД и авторизоваться под кредами, указанными в `.docker-compose/.env`
Допустимо выполнение любых команд. База тестовая, её можно ломать и не возвращать в исходное состояние
### e2e тестирование
При изменениях, связанных с фоновыми процессами, необходимо проводить e2e тестирование. Например, выполнение действия, которое инициирует фоновый процесс, в результате которого в БД должны появиться новые строки. Необходимо самостоятельно инициировать такое действие и проконтролировать конечный результат
### Репорты
После успешного прохождения QualityGate должен быть написан отчёт в папке `docs/reports` на основе `TEMPLATE.md`
