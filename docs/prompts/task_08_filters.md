# Контекст
Мы продолжаем работу в сервисе `vacancy_service` и теперь нам необходимо организовать систему хранения фильтров для поиска вакансий пользователя
# Пайплайн пользователя
1. Пользователь может воспользоваться фильтрами для поиска вакансий без сохранения
2. Пользователь может сохранить/переименовать или удалить свои пресеты фильтров
3. Пользователь может выполнить поиск вакансий по сохранённым пресетам фильтров
# Дополнительно
1. Фильтры мы привязываем к пользователю HH по id (хранит profile-service относительно id пользователя в системе). 
2. При удалении пользователя из нашей системы фильтры не удаляем!
3. При проектировании нарисуй схемы потока данных
# Какие фильтры нужны
Необходимо поддержать все фильтры, который доступны в качестве QuaryParams метода **GET** `/vacancies`. Нам нужно хранить фильтры не в JSON поле, а сделать в БД нормализованную модель. Критически важно использовать только те фильтры, что не находятся в статусе **DEPRECATED**. Пресет фильтров должен содержать пользовательское наименование (max=63)
# Main Backend
Нужны прокси методы к сервису вакансий LIST, GET, POST, PATCH, DELETE для вывода фильтров пользователя и управления ими. Стандартная сортировка по `-created_at`
LIST должен возвращать только список из наименований, а GET уже всю информацию со всеми полями
# Main Frontend
## Общая информация
На главной странице у нас расположен весь инструментарий работы с вакансиями. Есть разделение на две области. Слева вывод вакансий, справа вывод деталей вакансии. Сейчас мы будем работать в левой области и добавим дополнительный Drawer для настройки фильтров
Давай сделаем в левой области только кнопку "Фильтры" (иконкой). По её нажатию слева откроется Drawer
## Функционал Drawer
Drawer фильтров будет включать 3 области:
- работа с пресетами фильтров
- фильтры
- кнопки
### Работа с пресетами фильтров
Представляет собой кнопку, при нажатии на которую должно открываться всплывающее меню со списком сохранённых пресетов фильтров. По аналогии с селектором профиля HH в шапке сайта.
#### Поиск
Сверху выпадающего меню должен быть input, который выполняет поиск по регистронезависимому регулярному выражению. Поиск должен происходить исключительно на backend
#### Элемент пресета
Должен включать наименование пресета, кнопку редактирования наименования и кнопку удаления
#### Редактирование наименования
При нажатии на кнопку редактирования наименования открывается модальное окно с предзаполненным input. При нажатии на кнопку сохранения, наименование пресета изменяется. Должно открываться то же модальное окно, что и при сохранении пресета (описано далее), только в режиме изменения
#### Удаление
При нажатии на кнопку удаления должно открываться модальное окно с подтверждением опасного действия
#### Выбор пресета
При нажатии на пресет, форма должна заполниться значениями из этого пресета
#### Отсутствие пресета
Должен быть служебный пункт, который выбирает отсутствие пресета. При выборе такого пункта форма должна очиститься
### Область фильтрации
Представляет собой форму, содержащую поля всех доступных (согласно модели и API, не deprecated) фильтров
#### Фильтры-справочники
Есть некоторые поля фильтрации, которые представляются в виде справочников. Система управления справочниками была реализована в задаче `task_07_vacancy_dictionaries.md`. Изучи diff, артефакты и прочее для корректной реализации. Поля-справочники должны быть представлены в виде select (или multi-select если поддерживается) и запрашиваться с backend. Обязательным условием является наличие поисковой строки, если элементов более 5-ти. Поиск происходит на backend
### Область кнопок
После фильтров должны быть 3 кнопки:
- сброс фильтров
- сохранить фильтры
- применить фильтры
Давай расположим их в 2 ряда. В первом ряду будет сброс и сохранение, во втором ряду применение
#### Кнопка сброса
**Логика работы:**
При нажатии на кнопку сброса все поля формы очищаются
Сбрасывается выбранный пресет

**Параметры отображения:**
- **Цвет**: danger
- **Иконка**: фильтр (или перечёркнутый фильтр)
- **Положение иконки**: слева
- **Текст**: сброс
- **Подсказка при наведении**: Сбросить фильтры
#### Кнопка сохранения
**Логика работы:**
Логика работы зависит от того, выбран ли пресет фильтров или нет. Если пресет фильтров выбран, то при нажатии на кнопку сохранения происходит обновление пресета. При успехе Toast с уведомлением о том, что пресет "<наименование>" успешно сохранён
Если пресет не был выбран, то должно открыться модальное окно с input, в который пользователю необходимо ввести наименование нового пресета и только потом в модальном окне нажать на кнопку сохранения. При успехе toast

**Параметры отображения:**
- **Цвет**: secondary
- **Иконка**: дискета
- **Положение иконки**: слева
- **Текст**: сохранить
- **Подсказка при наведении**: Сохранить фильтры
#### Кнопка применения
**Логика работы:**
Пока не реализуем никакую логику. Будем реализовывать в следующей итерации

**Параметры отображения:**
- **Цвет**: primary
- **Иконка**: поиск, search, лупа
- **Положение иконки**: слева
- **Текст**: применить
- **Подсказка при наведении**: Применить фильтры
