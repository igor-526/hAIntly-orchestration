# Контекст
Нам нужно разработать микросервис `profile-service`. Он будет отвечать за вход пользователя в HH
# Протокол нового микросервиса
### Шаблон
В папке `fastapi_template` есть шаблон пустого микросервиса без авторизации и прочего
Если мы создаём новый микросервис (а этот как раз тот случай), то нам необходимо скопировать этот шаблон в целевой сервис папки `services`
### Контейнеризация
В папке `.docker-compose` есть файлы, с помощью которых поднимаются все сервисы:
- `.docker-compose/docker-compose.be.yml` - основной Backend и миграции БД. Все микросервисы должны быть под name `haintly`
- `.docker-compose/docker-compose.fe.yml` - frontend
- `.docker-compose/docker-compose.infra.yml` - инфраструктура (всё обслуживающее и БД)
### Database credentials
Так как БД поднимается в инфраструктурном docker-compose, то и файл .env находится в папке с этими файлами.
Именование переменных и их значения должны происходить по следующему шаблону:
```plaintext
<SERVICE>_POSTGRES_USER=<service>_user
<SERVICE>_POSTGRES_PASSWORD=<service>_password
<SERVICE>_POSTGRES_DB=<service>_db
<SERVICE>_POSTGRES_EXP_PORT=5451
```
Например, `PROFILE_POSTGRES_USER=profile_user`
### Makefile
В корневом Makefile должны быть следующие команды:
- `<service>` (be, fe, profile)- запускает docker compose выбранного сервиса
- `<service>-build` (be-build, fe-build, profile-build) - собирает образы выбранного сервиса с нуля
### Порты
Все сервисы должны пробрасывать наружу порты. Последняя цифра порта для сервиса, БД и тд должна быть одинаковая. Для FastAPI это 81xx, для PostgreSQL это 54xx, для frontend это 31xx
Например, main-be. API 81**01**, БД 54**01**, фронт 31**01**
profile-service. API 8102, БД 5452, фронт не требуется
### Авторизация
Авторизация микросервисов не требуется, так как микросервисы будут работать исключительно в локальной сети и к ним будет обращаться только основной Backend
# Авторизация основного Backend
При общении микросервисы могут слать запросы к основному Backend без авторизации
Давай реализуем такую авторизацию для микросервисов на основном Backend
Для этого на основном Backend сделаем сервисный ключ. Микросервисы будут передавать его в заголовках и это будет заменять авторизацию по Cookies. Для идентификации пользователя будем использовать заголовок `X-User-Id` как на стороне основного backend, так и на стороне микросервисов
# Задачи по микросервису
Нам нужно реализовать хранение пользователей и их авторизацию. Хранить пользователей HH будем в БД. Храним всю информацию о пользователе, которую предоставляет HH (именно профиль, не связанные сущности)
Access и refresh токены обязательно необходимо хранить в зашифрованном виде. Шифруем перед записью, расшифровываем после чтения налету
Нам необходимо в .env добавить переменную `HH_TOKEN_ENCRYPT_KEY`, которая будет хранить ключ шифрования
Необходим следующий функционал:
- авторизация через HH (комбинация методов для получения ссылки на вход, получение токенов)
- получение аккаунтов пользователя (LIST и GET методы. Пользователя необходимо передавать из основного backend через заголовок)
- отвязка аккаунта (удаление)
Подумай над формированием state для определения пользователя системы. Для авторизации нам необходимо будет делать служебную страницу на фронте, которая будет направлять запрос. Эта страница не должна быть защищена авторизацией. Заложи в state информацию о пользователе, к которому привязываем пользователя. Я говорю не о той странице, с которой происходит инициализация входа, а о той, на которую произойдёт redirect
# Задачи для основного main-be
### Проксирование
Так как frontend может обращаться только к основному backend, нам нужно реализовать для микросервиса client прокси к этому сервису с возможностью прокидывания авторизованного пользователя
Нам нужны методы на авторизацию, получение аккаунтов пользователя, комбинация методов для входа в hh и удаление аккаунта
### Метод авторизации
После редиректа на служебную страницу на фронте, она сделает запрос на backend. Этот метод не должен быть защищён авторизацией. Информация о пользователе, к которому привязывается аккаунт находится в state
### Мультиаккаунт
Проект должен поддерживать мультиаккаунт HH. То есть к одному профилю в системе может быть привязано несколько аккаунтов HH. Для этого нам необходимо модифицировать модель пользователя полем, которое указывает на то, какой аккаунт сейчас используется. Лучше сделать это уникальным идентификатором записи в БД микросервиса
# Задачи для main-fe
### Действие входа в HH
При запуске действия входа в HH необходимо открывать небольшое окошко, в котором сначала будет UI HH для входа, а потом служебная страница, с которой будет делаться запрос. После успешной авторизации это мини окошко закрывается
### Макет основной страницы
Я хочу разместить основные элементы управления и взаимодействия на одной главной странице Визуально мы будем делить весь интерфейс на 3 зоны
#### Шапка и другие элементы
Занимает всю ширину и находится наверху
В шапке нам нужно разместить следующее:
- логотип (пока что это просто текст HAIntly)
- профиль, совмещенный с меню (в меню пока добавим настройки и выход из аккаунта)
- селектор аккаунта HH.
Селектор аккаунта HH пока что самое важное, что нам нужно проработать Он будет выполнен в форме dropdown блока, в котором пользователю доступны следующие элементы управления: 
- добавление нового аккаунта HH
- выбор аккаунта
- удаление аккаунта
Полное снятие выбора аккаунта не допустимо по UX, но такая ситуация может произойти (например, сразу после регистрации или после удаления последнего аккаунта) При таком сценарии кнопка активации dropdown меню сменяется на кнопку входа в новый аккаунт HH. Если у пользователя не выбран ни один аккаунт, вместо блоков ниже должна быть надпись о том, что для продолжения работы необходимо выполнить вход в аккаунт HH
#### Рабочая область
Она занимает место до конца экрана и делится на 2 вертикальные области. Слева вакансии и их поиск. Справа область работы с выбранной вакансией
#### Вакансии и их поиск 
- кнопка выбора фильтров по вакансии
- селектор резюме профиля HH (текст)
- список самих вакансий
-Вакансия в списке состоит из следующих элементов: наименование, которое может быть длинным; зарплата; индекс релевантности (Цифра 0-100 или значок отсутствия анализа)
#### Область работы с выбранной вакансией
Тут только информация о ванасии и функциональные кнопки по типу "AI сопроводительное", "AI процент релевантности" Левая и правая области имеют минимальную ширину. Ширина может быть выбрана пользователем методом DnD перегородки областей

# Redirect URL
На данный момент redirect настроен на `http://localhost:3101`
Redirect URL должен настраиваться параметрами окружения `.env`

К примеру, если в настройках сохранен `http://example.com/oauth`, то **разрешено** указывать:
- `http://www.example.com/oauth` — поддомен (www);
- `http://www.example.com/oauth/sub/path` — уточнение пути;
- `http://example.com/oauth?lang=RU` — дополнительный параметр;
- `http://www.example.com/oauth/sub/path?lang=RU` — всё вместе.

**Запрещено**:
- `https://example.com/oauth` — различные протоколы;
- `http://wwwexample.com/oauth` — различные домены;
- `http://test.example.com/` — дополнительный поддомен;
- `http://wwwexample.com/` — другой путь;
- `http://example.com/oauths` — другой путь;
- `http://example.com:80/oauths` — указание изначально отсутствующего порта;
