# Архитектура сервисов HAIntly

## Общая схема

HAIntly строится как набор сервисов с раздельным владением данными.

```text
Пользователь
    |
    | HTTP / SSE
    v
 main-fe <----------> main-be
                         |
                         | HTTP
              +----------+----------+------------------+
              v                     v                  v
      profile-service       vacancy-service       ai-service
                                    |                  ^
                                    +--- NATS JS ------+
                                    |
                                    +--- NATS JS ---> notification-service
                                                           |
                                                           +--- NATS JS ---> main-be
```

- HTTP используется для синхронных команд и чтения.
- SSE используется `main-be` для доставки обновлений в `main-fe`.
- NATS JetStream используется для асинхронных заданий и событий.
- Каждый backend-сервис владеет собственной PostgreSQL БД.
- Прямой доступ к БД другого сервиса запрещён.

Точные endpoint, NATS subjects, event schemas, порты и параметры deployment фиксируются отдельными OpenSpec changes.

## Пользовательский поток

1. Пользователь регистрируется, подтверждает email и входит в HAIntly.
2. Подключает один или несколько HH-аккаунтов по OAuth.
3. Выбирает доступное резюме для поиска.
4. Запускает фоновый сбор вакансий и при необходимости повторяет его.
5. Запускает AI-оценку вакансий относительно выбранного резюме.
6. Сортирует вакансии по оценке от 1 до 100.
7. Генерирует сопроводительное письмо для конкретной вакансии.
8. Скрывает вакансию навсегда либо отмечает, что уже откликнулся.
9. Получает уведомления о фоновых операциях через SSE.

## `main-fe`

**Технология:** Next.js.

**Ответственность:** UX, UI и отображение пользовательских уведомлений.

**Связи:**

- HTTP → `main-be` для команд и данных;
- SSE ← `main-be` для уведомлений и статусов длительных операций.

Frontend не обращается напрямую к HH, AI-провайдеру или профильным backend-сервисам.

## `main-be`

**Технологии:** FastAPI, PostgreSQL, SSE.

**Ответственность:**

- регистрация и подтверждение email;
- авторизация и аутентификация пользователей;
- управление пользователями;
- публичный API для `main-fe`;
- маршрутизация и оркестрация профильных сервисов;
- доставка уведомлений в UI по SSE.

**Связи:**

- HTTP ↔ `main-fe`;
- SSE → `main-fe`;
- HTTP → `profile-service` для HH-профилей и резюме;
- HTTP → `vacancy-service` для вакансий, парсинга и статусов;
- HTTP → `ai-service` для управления AI-настройками;
- HTTP → `notification-service` для истории и настроек уведомлений;
- NATS JetStream ← `notification-service` для новых уведомлений.

## `profile-service`

**Технологии:** FastAPI, PostgreSQL, aiohttp, cryptography.

**Ответственность:**

- связь пользователей HAIntly с несколькими HH-аккаунтами;
- HH OAuth;
- защищённое хранение и обновление HH access/refresh token;
- получение и хранение доступных резюме;
- выбранное резюме для поиска вакансий.

**Связи:**

- HTTP ← `main-be` для OAuth и управления профилями;
- HTTP ← `vacancy-service` для контролируемого получения доступа к HH.

HH-токены не передаются в frontend и не становятся данными других сервисов.

## `vacancy-service`

**Технологии:** FastAPI, PostgreSQL, aiohttp, Celery, NATS JetStream.

**Ответственность:**

- фоновый сбор и повторный сбор вакансий;
- хранение вакансий и пользовательских статусов;
- выдача вакансий;
- скрытие вакансии навсегда;
- ручная отметка об отклике;
- постановка AI-задач на оценку и генерацию письма;
- публикация событий для уведомлений.

**Связи:**

- HTTP ← `main-be` для управления вакансиями и сбором;
- HTTP → `profile-service` для доступа к HH от имени подключённого аккаунта;
- NATS JetStream ↔ `ai-service` для заданий и результатов;
- NATS JetStream → `notification-service` для пользовательских уведомлений.

Celery отвечает только за внутреннюю фоновую обработку. Его broker/backend должны быть определены в отдельном техническом design.

## `ai-service`

**Технологии:** FastAPI, PostgreSQL, Celery, клиент OpenAI-совместимого API для OpenRouter.

**Ответственность:**

- оценка релевантности вакансии выбранному резюме от 1 до 100;
- генерация сопроводительных писем;
- управление моделями и промптами;
- статистика потребления токенов.

**Связи:**

- HTTP ← `main-be` для административных AI-настроек;
- NATS JetStream ↔ `vacancy-service` для задач и результатов;
- HTTPS → OpenRouter для AI-запросов.

Взаимодействие с провайдером изолируется адаптером. Персональные данные в prompts и responses защищаются согласно OpenSpec.

## `notification-service`

**Технологии:** FastAPI, PostgreSQL, Celery, NATS JetStream.

**Ответственность:**

- история пользовательских уведомлений;
- настройки уведомлений;
- формирование пользовательских и технических сообщений;
- доставка новых уведомлений в `main-be`.

**Связи:**

- HTTP ← `main-be` для истории и настроек;
- NATS JetStream ← `vacancy-service` для событий фоновых процессов;
- NATS JetStream → `main-be` для доставки готового уведомления в UI.

Сервис не поддерживает SSE-соединения с браузером: эта граница принадлежит `main-be`.

## Архитектурные инварианты

- `main-fe` знает только публичный контракт `main-be`.
- Межсервисные изменения контрактов сначала отражаются в OpenSpec.
- Сервис не читает и не изменяет таблицы другого сервиса.
- Токены HH принадлежат `profile-service` и шифруются в покое.
- Вакансии и пользовательские статусы принадлежат `vacancy-service`.
- AI score и письма вычисляются `ai-service`, но инициируются в контексте вакансии.
- История и настройки уведомлений принадлежат `notification-service`.
- Повторная доставка асинхронных сообщений учитывается при проектировании consumers.
