# SMOKE-тесты на реальном API

Этот скилл запускает SMOKE-тесты из плана с автоматической авторизацией через cookie.

## Аргументы

- **роль** (опционально): `admin` или `user`. По умолчанию `admin`.
- **план** (опционально): путь к файлу плана. По умолчанию ищет активный план в `docs/plans/`.

Примеры вызова:
- `/smoke` — запуск как admin, план выбирается из контекста
- `/smoke user` — запуск как user
- `/smoke admin docs/plans/feature-nex-1079-1-upravlenie-pokazatelyami.md` — явный план

## Шаг 1: Прочитай credentials

Прочитай файл `.claude/skills/api-smoke-test/credentials.json`. Из него возьми:
- `base_url` — базовый URL API
- `auth_endpoint` — путь для логина
- `cookie_jar` — путь к файлу куков
- `roles.<роль>.login` и `roles.<роль>.password` — данные для входа

## Шаг 2: Найди SMOKE-тесты в плане

Если план не передан аргументом — определи его из контекста разговора (последний упомянутый план).

Прочитай файл плана. Найди секцию `### SMOKE-тесты на реальном API` (или аналогичную — `SMOKE`, `Smoke tests`, `smoke`).

Из секции извлеки:
- **Переменные** — блок `BASE_URL`, `*_ID` и т.д. (обычно в ```-блоке сразу после заголовка)
- **Таблицу тестов** — строки вида `| SM-01 | запрос | проверка |`

Если план не найден и не определяется из контекста — сообщи пользователю и попроси уточнить.

## Шаг 3: Авторизуйся

> ⚠️ API использует **cookie-авторизацию**, не Bearer-токен. Все запросы должны идти с параметром `-b <cookie_jar>`.

Выполни логин и сохрани куки:

```bash
curl -s -o /dev/null -w "%{http_code}" \
  -c <cookie_jar> \
  -X POST "<base_url><auth_endpoint>" \
  -H "Content-Type: application/json" \
  -d '{"login": "<login>", "password": "<password>"}'
```

Ожидаемый статус: `204`. Если получен другой код — сообщи об ошибке авторизации и остановись.

После успешного логина можно также проверить себя:

```bash
curl -s -b <cookie_jar> "<base_url>/api/auth/verify"
```

Ответ должен содержать поле `username`.

## Шаг 4: Запусти тесты

Для каждого теста из таблицы:

1. Подставь переменные из блока переменных в URL и тело запроса
2. Сформируй `curl`-команду с флагом `-b <cookie_jar>`
3. Выполни запрос
4. Проверь ответ по условию из колонки «Проверка»

### Формат curl-запроса

**GET:**
```bash
curl -s -o /dev/null -w "%{http_code}\n%{response}" \
  -b <cookie_jar> \
  "<base_url><путь>"
```

Для получения тела ответа удобнее:
```bash
curl -s -b <cookie_jar> "<base_url><путь>" | python3 -m json.tool
```

**PATCH/POST с телом:**
```bash
curl -s -b <cookie_jar> \
  -X PATCH "<base_url><путь>" \
  -H "Content-Type: application/json" \
  -d '<тело>'
```

### Что проверять

| Условие в плане | Как проверить |
|---|---|
| `200, total=N` | `.total == N` в JSON-ответе |
| `200, len(items)=N` | `len(response["items"]) == N` |
| `validation error` | HTTP статус `422` |
| `400` / `404` / `401` | HTTP статус совпадает |
| поле обновлено | значение поля в ответе совпадает с отправленным |

## Шаг 5: Отчёт

По завершении всех тестов выведи итоговую таблицу:

```
| #     | Запрос                        | Статус | Результат        |
|-------|-------------------------------|--------|------------------|
| SM-01 | GET /                         | ✅     | total=78         |
| SM-02 | GET /?limit=1                 | ✅     | items содержат icon |
| SM-05 | GET /?search=частотность      | ❌     | ожидал total=6, получил total=4 |
```

Итог: `N/M тестов прошли`.

Если какой-то тест упал — выведи полный ответ API для диагностики.

## Нюансы

- **Порт**: По умолчанию `8000`. Если BE поднят на другом порту — `base_url` в `credentials.json` нужно поправить.
- **Idempotency**: Тесты с `PATCH` должны восстанавливать исходное состояние (обычно это явно указано в плане — тест `SM-17` и подобные).
- **Переменные плана**: Блок с `VID_ID`, `WS_ID` и т.д. всегда идёт сразу после заголовка SMOKE-секции — прочитай его до начала тестов.
- **Токен живёт 15 минут** — ровно столько сколько нужно на один прогон. Файл куков перезаписывается при каждом вызове скилла, сохранять его между сессиями не нужно.
