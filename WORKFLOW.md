# Workflow разработки HAIntly

## Общая схема

```text
Текст задачи или docs/prompts/<slug>.md
    ↓
openspec-propose
    ↓
Ревью артефактов человеком
    ↓
openspec-apply-change
    ├── Backend subagent
    └── Frontend subagent
    ↓
Quality Gate subagent
    ├── НА ДОРАБОТКУ → новые задачи → apply
    └── ОДОБРЕНО → sync specs → archive
```

Смысловое содержание ведётся на русском языке. Обязательные парсеру заголовки и ключевые слова OpenSpec остаются английскими.

## 1. Подготовка задачи

Задача передаётся текстом либо сохраняется в `docs/prompts/<slug>.md`. Полезный вход содержит проблему, ожидаемый результат, ограничения и известные acceptance criteria.

## 2. Propose

Router запускает `openspec-propose`. Change должен определить:

- цель и scope;
- затронутые сервисы;
- требования и сценарии;
- технические решения и потоки данных;
- атомарные задачи Backend, Frontend и Quality Gate.

Пользователь проверяет артефакты до реализации.

## 3. Apply и маршрутизация

Router запускает `openspec-apply-change`, получает `contextFiles` из CLI и передаёт профильному subagent:

- профиль `agents/backend.md` для backend-сервисов;
- профиль `agents/frontend.md` для `main-fe`;
- OpenSpec artifacts;
- связанные файлы и сервисные ограничения.

Независимые задачи могут выполняться параллельно только при непересекающихся write scopes. Профильный агент отмечает checkbox сразу после реализации и проверки задачи.

Если выяснилось, что design или spec неверны, сначала обновляются OpenSpec-артефакты. Исполнитель не принимает скрытое архитектурное решение в коде.

## 4. Quality Gate

После реализации Router запускает отдельный subagent с `agents/quality_gate.md`. Он проверяет diff, требования, сервисные границы, безопасность и реально доступные проверки.

Результат:

- `ОДОБРЕНО` — реализация готова к завершению change;
- `НА ДОРАБОТКУ` — замечания добавляются в `tasks.md`, после чего apply и Quality Gate повторяются.

Отчёт сохраняется в `docs/reports/<change-name>-review.md`, если change требует постоянного review-артефакта.

## 5. Завершение

После одобрения:

1. Синхронизировать delta specs через `openspec-sync-specs`, если требования изменились.
2. Убедиться, что все задачи отмечены завершёнными.
3. Архивировать change через `openspec-archive-change`.

Git commit, branch и PR выполняются только по отдельному запросу пользователя.

## Bootstrap

Первоначальная настройка или ремонт самого оркестратора может выполняться напрямую без change. Исключение ограничено `AGENTS.md`, профилями агентов, OpenSpec-конфигурацией и workflow-документацией; продуктовый код под него не подпадает.
